# 0505 新对话交接摘要

## 当前在干什么

当前主线是把场景 C 的飞书文档迭代能力收稳，然后在这个基础上继续补“富媒体与画布操作”。

这条线已经分成两部分：

1. 文本/章节类编辑先收口，重点解决意图识别、锚点定位、插入方向、后验校验这些稳定性问题。
2. 富媒体能力按 `lark-cli 1.0.23` 的真实能力做，先明确“每个语义对应哪条 CLI 命令”，再补执行链路，不再把富媒体伪装成文本 patch。

## 已经做了什么

### 一、文档文本/章节编辑主链路已做的事

- 已完成一轮围绕确定性锚点的重构，核心方向是让 section/block/media 定位更多依赖结构快照，而不是模糊文本猜测。
- `DocumentEditIntentResolver` 已补强章节编号、层级章节、前插入/后插入的规范化逻辑。
- `DocumentAnchorResolver` 已补强基于章节编号、层级路径、父章节作用域的确定性解析。
- `DocumentStructureSnapshot` / builder 已补强结构索引，支持更稳定的 section 定位。
- `DocumentPatchCompiler` 对 section before/after insert 的编译做过修正，避免不安全 move target。
- `DocumentTargetStateVerifier` 已放宽“插入成功但误判失败”的脆弱校验逻辑。

### 二、已确认修过的具体问题

- 修过“章节语义被错误降级成 quotedText/block 锚点”。
- 修过“`1.3` 这类章节编号被误当成顶层 ordinal”。
- 修过“后插入”被错误识别成 before-section insert。
- 修过“飞书已插入成功，但 verifier 仍报插入失败”。
- 修过用户 case `1.1 后插入一章1.2` 的归一化方向错误：
  - 现在应归一化为锚点 `1.1`
  - 动作应走 `INSERT_BLOCK_AFTER_ANCHOR`
  - 不应再把 `1.2` 当成被定位锚点

### 三、当前代码状态

- 富媒体语义枚举已经在代码里存在：
  - `INSERT_IMAGE_AFTER_ANCHOR`
  - `REPLACE_IMAGE`
  - `INSERT_TABLE_AFTER_ANCHOR`
  - `REWRITE_TABLE_DATA`
  - `APPEND_TABLE_ROW`
  - `INSERT_WHITEBOARD_AFTER_ANCHOR`
  - `UPDATE_WHITEBOARD_CONTENT`
  - `MOVE_MEDIA_BLOCK`
  - `RELAYOUT_SECTION`
- `DefaultDocumentIterationExecutionService` 已经接入双通道：
  - 文本类走 `DocumentPatchExecutor`
  - 富媒体类走 `RichContentExecutionPlanner` + `RichContentExecutionEngine`
- 但富媒体这条链还没有完全按 CLI 能力收口，尤其需要把“语义 -> CLI 命令”的映射做实。

### 四、已经确认的 CLI 能力边界

基于当前本机 `lark-cli 1.0.23`，已确认：

- 文档图片插入可对接 `docs +media-insert`
- 文档已有白板更新可对接 `docs +whiteboard-update`
- 文档媒体上传/预览/下载能力存在
- 没确认到文档侧完整的“细粒度表格样式编辑”能力
- 没确认到文档侧完整的“图片尺寸、环绕、裁剪、边框”等样式能力

因此当前结论是：

- 图片插入：可做稳定版
- 白板更新：可做稳定版
- 表格插入/整表重写：可以做基础版
- 细粒度样式调整：当前不该承诺
- 复杂版式布局：当前不该承诺，只能做块级/章节级重排

## 将要做什么

新对话窗口里，主任务应切到“富媒体与画布操作正式落地”，按下面顺序推进：

1. 先做能力矩阵
   - 明确每个语义动作到底落哪条 CLI 命令
   - 明确哪些语义当前只能降级成 approval-only，哪些可以真实执行

2. 再做执行方案
   - 图片插入优先
   - 白板更新第二优先
   - 表格插入/整表写入第三优先
   - 白板新建并插入，取决于当前 CLI/skill 是否已有 create + embed 链路

3. 最后再落代码
   - 补齐 rich execution plan 到具体 CLI command binding
   - 补齐执行后校验
   - 补齐针对图片/表格/白板的测试

## 新对话建议直接延续的工作方式

新窗口建议直接从这句话开始：

“先核对 `lark-cli 1.0.23` 下富媒体/画布相关命令，把每个语义动作映射到明确 CLI，再给出实现计划并开始改代码。”

## 当前最重要的判断

这轮工作不是“从零设计富媒体架构”，而是：

- 文本链路先收稳，已经基本到位
- 富媒体链路已有骨架
- 下一步要做的是把富媒体语义和 `lark-cli 1.0.23` 的真实命令能力一一对上，然后把执行链真正补通
