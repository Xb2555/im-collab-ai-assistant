  ## 工作方式

  - 默认一次做完：分析、改码、补必要校验、完成交付，不做半截工程。
  - 直接给出最优架构和最优实现，不保守，不留手，不拆成多轮试探。
  - 不做未明确要求的功能、重构、优化、文档、示例、说明。
  - 遇到关键信息缺失，只问一个最关键的问题。
  - 遇到不确定的点，先查官方资料；本地和官方资料都不能定论时再问用户，不允许靠猜。

  ## 输出要求

  - 只输出完成任务所必需的内容。
  - 不写废话，不客套，不解释动机，不重复用户原话。
  - 不主动输出 README、设计说明、使用说明、示例代码。
  - 不写测试代码，除非用户明确要求。
  - 不写注释，除非用户明确要求。
  - 如果只改一个函数，只返回该函数或只修改该函数，不扩散输出。

  ## 架构与代码约束

  - `common` 模块放公共工具、共享 entity、跨模块 contract、统一异常。
  - 业务异常统一使用 `common` 中的自定义业务异常，禁止在业务代码里散落通用运行时异常表达业务语义。
  - 业务模块只放本模块职责内的逻辑，不重复定义公共模型。
  - 代码命名、分包、分层遵守 Alibaba Java 开发规范。
  - 保持模块边界清晰，避免“大而全”服务类。
  - 不改未被要求的模块和逻辑，不顺手重构。

  ## 测试与验证

  - 只跑最关键、最核心的测试链路。
  - 不随意扩大测试范围，不生成额外测试。
  - Maven 测试若涉及依赖下载、插件下载或外网解析，默认需要网络/提权权限。
  - 遇到 Maven 因沙箱或联网限制失败，直接申请执行权限，不先空谈。

  ## 权限策略

  - 本地能确认的先本地确认。
  - 官方资料能确认的先查官方资料。
  - 需要联网、下载依赖、运行受限命令时，直接申请最小必要权限。
  - 不绕过权限，不猜测环境状态。

  ## 交付标准

  - 以可直接落地、可直接运行、可直接合并为标准。
  - 不做半成品，不把剩余工作留给用户补。
  - 除非明确阻塞，否则不在中途停下问“要不要继续”。

## PPT 与 CLI 踩坑约束

- 在当前 Windows 环境下，`lark-cli slides +create --slides` 传大段 XML JSON 极不稳定，`--slides` 的值会被本地 CLI 可执行链拆成位置参数；生成链路不要依赖这条命令承载真实生产页面内容。
- 生成 PPT 的稳定链路是：先 `slides +create` 创建空白演示文稿，再逐页 `xml_presentation.slide.create`；涉及图片时，先 `slides +media-upload` 拿到真实 `file_token`，再写入 `<img src="boxcn...">`。
- `slides +media-upload --file` 的参数是本地文件路径，不是 `@path`；传 `@./...` 会报 `file not found`。
- `xml_presentation.slide.create` 的 `xml_presentation_id` 必须来自真实 slides 演示文稿；新建后立即写页可能命中 `3350002 not found`，需要做短重试和就绪等待。
- 在当前仓库里，PPT 生成用到的下载图片统一落在仓库根目录 `.ppt-generated-assets/` 下；不要再写到系统临时目录，否则后续上传或路径校验容易失败。
- 图片搜索节点只允许返回可直接下载的原始资源 URL，不允许返回详情页、搜索页、跳转页、`undraw.co/api/...` 这类不稳定接口 URL。
- 图片下载必须做候选兜底：单个任务至少准备多个候选 URL，前一个失败就继续尝试下一个，否则 PPT 会退化成接近纯文本页面。
