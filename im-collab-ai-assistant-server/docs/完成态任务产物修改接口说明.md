# 完成态任务产物修改接口说明

本文面向前端说明 GUI 侧如何对已完成任务继续调整，包括原地修改已有 PPT / DOC 产物，以及保留旧产物后重新规划并重跑任务。

## 1. 功能边界

已完成任务现在可以继续发起 `REPLAN`。

前端统一调用：

```http
POST /api/planner/tasks/{taskId}/commands
```

后端会根据用户反馈和产物策略分流：

> 当前已支持两类完成态原地修改：
> 1. `PPT` 原地修改
> 2. `DOC` 原地修改
>
> 其中 `DOC` 可能因为风险较高进入待确认状态，此时前端仍统一走 `/commands` 续交互，不需要单独调用 `document-iteration/approval`。

| 场景 | 后端行为 | 是否新 taskId | 是否新增产物 |
| --- | --- | --- | --- |
| 明确修改已有 PPT，如“把第 1 页标题改成 xxx” | 原地编辑指定 PPT | 否 | 否 |
| 明确修改已有 DOC，如“在 1.2 后补一段风险分析” | 原地编辑指定 DOC | 否 | 否 |
| DOC 修改命中高风险场景 | 返回待确认修改计划，等待用户确认/修改/拒绝 | 否 | 否 |
| 保留旧版、另做一版、整体重新规划、重新生成 | 重新进入 Planner，生成新计划 | 否 | 执行后新增产物 |
| 已完成任务只有一个可编辑产物，且用户明确修改内容 | 默认修改该产物 | 否 | 否 |
| 已完成任务有多个可编辑产物但未指定产物 | 返回追问，让用户选择产物 | 否 | 否 |

> 注意：`REPLAN` 只负责调整计划或触发原地修改。整体重新规划后通常会回到 `PLAN_READY`，前端仍需再调用 `CONFIRM_EXECUTE` 才会真正重跑。

## 2. 前端判断入口

查询运行态：

```http
GET /api/planner/tasks/{taskId}/runtime
```

当返回：

```json
{
  "data": {
    "task": {
      "status": "COMPLETED"
    },
    "actions": {
      "canReplan": true
    }
  }
}
```

说明该完成态任务允许继续调整。

## 3. 请求字段

接口：

```http
POST /api/planner/tasks/{taskId}/commands
Content-Type: application/json
```

请求体：

```json
{
  "action": "REPLAN",
  "feedback": "把第1页标题改成采购风险与建议",
  "artifactPolicy": "EDIT_EXISTING",
  "targetArtifactId": "artifact-id",
  "version": 3
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `action` | string | 是 | `REPLAN`、`CONFIRM_EXECUTE`、`RESUME`、`CANCEL`、`RETRY_FAILED`。 |
| `feedback` | string | 否 | `REPLAN` 和 `RESUME` 时通常需要。建议写完整自然语言，不要只传枚举。 |
| `artifactPolicy` | string | 否 | 产物策略。旧前端不传时按 `AUTO` 处理。 |
| `targetArtifactId` | string | 否 | 指定要修改的已有产物 ID。多产物场景建议传。 |
| `version` | number | 是 | 当前任务版本号，用于并发冲突校验。可从 `/runtime.data.task.version` 或 `GET /{taskId}` 获取。 |

`artifactPolicy` 可选值：

| 值 | 含义 | 推荐使用场景 |
| --- | --- | --- |
| `AUTO` | 让后端根据用户话术自动判断 | 旧逻辑兼容或不确定场景 |
| `EDIT_EXISTING` | 倾向修改已有产物 | 用户明确“修改这个 PPT / 文档” |
| `CREATE_NEW` | 倾向创建新产物 | 用户明确要新的产物形式 |
| `KEEP_EXISTING_CREATE_NEW` | 保留旧产物并新建一版 | 用户明确“保留旧版 / 另做一版 / 重新生成” |

建议前端同时传 `artifactPolicy` 和明确的 `feedback`。例如不要只传：

```json
{
  "artifactPolicy": "KEEP_EXISTING_CREATE_NEW"
}
```

推荐：

```json
{
  "artifactPolicy": "KEEP_EXISTING_CREATE_NEW",
  "feedback": "保留现有文档，整体重新规划并生成一份新版复盘文档，补充风险分析章节"
}
```

## 4. 原地修改已有 PPT

适用条件：

1. 任务状态为 `COMPLETED`。
2. 任务已有 PPT artifact。
3. 用户明确说明要改哪一页、改成什么内容。
4. 如果存在多个 PPT，前端最好传 `targetArtifactId`。

请求示例：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "REPLAN",
    "feedback": "把第1页标题改成采购风险与建议",
    "artifactPolicy": "EDIT_EXISTING",
    "targetArtifactId": "59e8550f-12ee-4089-ab3e-a231983c32d4",
    "version": 3
  }'
```

典型响应：

```json
{
  "code": 0,
  "data": {
    "taskId": "task-id",
    "version": 3,
    "planningPhase": "COMPLETED",
    "assistantReply": "已修改 PPT 第 1 页：采购风险与建议",
    "actions": {
      "canConfirm": false,
      "canReplan": true,
      "canCancel": false,
      "canResume": false,
      "canInterrupt": false,
      "canRetry": false
    }
  },
  "message": "ok"
}
```

修改完成后，前端应再查 `/runtime` 刷新产物状态：

```http
GET /api/planner/tasks/{taskId}/runtime
```

产物会保持同一个 `artifactId` 和 `url`，不会新增同类型产物：

```json
{
  "data": {
    "task": {
      "status": "COMPLETED"
    },
    "artifacts": [
      {
        "artifactId": "59e8550f-12ee-4089-ab3e-a231983c32d4",
        "type": "PPT",
        "title": "原 PPT 标题",
        "url": "https://xxx.feishu.cn/slides/xxx",
        "preview": "已修改 PPT 第 1 页：采购风险与建议",
        "status": "UPDATED"
      }
    ]
  }
}
```

## 5. 原地修改已有 DOC

适用条件：

1. 任务状态为 `COMPLETED`。
2. 任务已有 DOC artifact。
3. 用户明确说明要改哪个章节、补什么内容、删除什么内容，或插入什么内容。
4. 如果存在多个 DOC，前端最好传 `targetArtifactId`。

请求示例：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "REPLAN",
    "feedback": "在 1.2 后补充一段风险分析",
    "artifactPolicy": "EDIT_EXISTING",
    "targetArtifactId": "doc-artifact-id",
    "version": 3
  }'
```

### 5.1 直接执行成功

低风险或可直接执行的文档修改会直接返回完成态：

```json
{
  "code": 0,
  "data": {
    "taskId": "task-id",
    "version": 3,
    "planningPhase": "COMPLETED",
    "assistantReply": "已补充风险分析章节",
    "actions": {
      "canConfirm": false,
      "canReplan": true,
      "canCancel": false,
      "canResume": false,
      "canInterrupt": false,
      "canRetry": false
    }
  },
  "message": "ok"
}
```

前端应再查 `/runtime` 刷新同一个 DOC artifact 的 `preview/status`。

### 5.2 进入待确认修改计划

高风险 DOC 修改会先返回待确认状态：

```json
{
  "code": 0,
  "data": {
    "taskId": "task-id",
    "version": 3,
    "planningPhase": "ASK_USER",
    "assistantReply": "已生成待确认的文档修改计划",
    "actions": {
      "canConfirm": true,
      "canReplan": true,
      "canCancel": false,
      "canResume": true,
      "canInterrupt": false,
      "canRetry": false
    }
  },
  "message": "ok"
}
```

这类状态的含义是：

1. `canConfirm=true`：用户接受当前修改计划，可直接确认执行。
2. `canResume=true`：用户要补充修改意见，或明确拒绝本次修改。
3. 前端继续统一调用 `/commands`，不要再调用独立的 `document-iteration/approval`。

#### 确认执行当前 DOC 修改计划

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "CONFIRM_EXECUTE",
    "version": 4
  }'
```

典型响应：

```json
{
  "code": 0,
  "data": {
    "taskId": "task-id",
    "planningPhase": "COMPLETED",
    "assistantReply": "文档修改已执行",
    "actions": {
      "canConfirm": false,
      "canReplan": true,
      "canCancel": false,
      "canResume": false,
      "canInterrupt": false,
      "canRetry": false
    }
  },
  "message": "ok"
}
```

#### 带补充意见继续修改计划

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "RESUME",
    "feedback": "请把风险分析改成整章重写，并保留原有结论",
    "version": 4
  }'
```

如果新的修改意见仍然需要确认，后端会继续返回 `planningPhase=ASK_USER`，并保持 `canConfirm=true`、`canResume=true`。

#### 拒绝本次 DOC 修改

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "RESUME",
    "feedback": "这次先不用改了",
    "version": 4
  }'
```

后端会结束这次文档修改挂起态，原文档保持不变，任务回到 `COMPLETED`。

## 6. 保留旧产物并重新规划重跑

适用条件：

1. 用户明确说“保留旧版”“另做一版”“重新生成”“整体重新规划”“重跑”。
2. 最终希望新增产物，而不是覆盖现有 PPT / DOC。

请求示例：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "REPLAN",
    "feedback": "保留现有文档，整体重新规划并生成一份新版文档，补充风险分析章节",
    "artifactPolicy": "KEEP_EXISTING_CREATE_NEW",
    "version": 3
  }'
```

典型响应：

```json
{
  "code": 0,
  "data": {
    "taskId": "task-id",
    "planningPhase": "PLAN_READY",
    "actions": {
      "canConfirm": true,
      "canReplan": true,
      "canCancel": true,
      "canResume": false,
      "canInterrupt": false,
      "canRetry": false
    }
  },
  "message": "ok"
}
```

前端此时应展示更新后的计划，并等待用户确认。用户确认后调用：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "CONFIRM_EXECUTE",
    "version": 4
  }'
```

执行完成后再查 `/runtime`，应看到旧产物仍在，并新增一个新的同类 artifact。

## 7. 多产物选择

如果同一个完成态任务下有多个可修改产物，且前端没有传 `targetArtifactId`，后端会进入追问态。

典型响应：

```json
{
  "code": 0,
  "data": {
    "planningPhase": "ASK_USER",
    "assistantReply": "这个任务下有多个可修改产物，你想修改哪一个？\n1. [PPT] 采购评审初版 965fcaac\n2. [DOC] 采购评审文档 e5e34273\n回复编号即可。",
    "actions": {
      "canResume": true
    }
  },
  "message": "ok"
}
```

GUI 侧有两种处理方式：

1. 推荐：前端自己展示 artifacts，让用户选择后重新发 `REPLAN`，并带上 `targetArtifactId`。
2. 兼容：直接调用 `RESUME`，把用户选择的编号传给后端。

`RESUME` 示例：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "RESUME",
    "feedback": "2",
    "version": 4
  }'
```

如果编号不存在，后端会继续返回提示，不会误改产物。

## 8. 关键响应字段

### `PlanPreviewVO`

`POST /commands` 返回 `PlanPreviewVO`：

| 字段 | 说明 |
| --- | --- |
| `taskId` | 当前任务 ID。完成态调整不会生成新 taskId。 |
| `version` | 当前任务版本。后续命令需要带最新版本。 |
| `planningPhase` | 当前阶段，如 `PLAN_READY`、`ASK_USER`、`EXECUTING`、`COMPLETED`。 |
| `cards` | 当前计划卡片。重新规划后会更新。 |
| `actions` | 前端可执行动作。DOC 待确认修改时通常会同时返回 `canConfirm=true` 和 `canResume=true`。 |
| `assistantReply` | 后端给用户看的提示，如“已修改 PPT 第 1 页”“已生成待确认的文档修改计划”。 |
| `accepted` | 是否被 Agent 接收为真实任务。 |
| `runtimeAvailable` | 是否可以查 `/runtime`。 |
| `transientReply` | 是否只是提示型回复，不应进入工作台。 |
| `capabilityHints` | 当前计划相关能力提示。 |

### `TaskDetailVO`

`GET /runtime` 返回 `TaskDetailVO`：

| 字段 | 说明 |
| --- | --- |
| `task` | 任务摘要，含 `status/version/progress`。 |
| `steps` | 当前步骤列表。 |
| `artifacts` | 产物列表，前端主要从这里拿 `artifactId/url/status/preview`。 |
| `events` | 时间线事件。 |
| `actions` | 当前运行态可执行动作。 |

### `TaskArtifactVO`

| 字段 | 说明 |
| --- | --- |
| `artifactId` | 产物 ID。原地编辑时保持不变。 |
| `type` | 产物类型，如 `PPT`、`DOC`。 |
| `title` | 产物标题。 |
| `url` | 飞书产物链接。原地编辑时保持不变。 |
| `preview` | 最近一次修改摘要。 |
| `status` | `CREATED`、`UPDATED` 等。 |
| `createdAt` | 创建时间。 |

## 9. 前端推荐交互

完成态任务详情页：

1. 先查 `/runtime`。
2. 如果 `actions.canReplan=true`，展示“调整任务 / 修改产物”入口。
3. 如果用户选择“修改已有产物”，前端展示 artifact 列表，要求用户选择具体产物。
4. 对 PPT 修改，要求用户输入明确修改内容，如“第几页”“改成什么”。
5. 对 DOC 修改，要求用户输入明确修改内容，如“哪个章节”“补什么内容”“删什么内容”。
6. 对 `PPT` 或 `DOC` 原地修改，都调 `REPLAN + EDIT_EXISTING + targetArtifactId`。
7. 如果返回 `planningPhase=ASK_USER` 且 `canConfirm=true`、`canResume=true`，按“待确认文档修改计划”处理：
   - 用户接受当前方案：调 `CONFIRM_EXECUTE`
   - 用户要补充修改意见：调 `RESUME`
   - 用户拒绝本次修改：调 `RESUME`，`feedback` 写明“先不用改了”“取消这次修改”等
8. 响应成功后立刻刷新 `/runtime`，展示同一个链接和更新后的 `preview`。

完成态整体重跑：

1. 用户选择“保留旧版并重新生成”。
2. 调 `REPLAN + KEEP_EXISTING_CREATE_NEW`，`feedback` 中写明新产物要求。
3. 若返回 `PLAN_READY`，展示新计划。
4. 用户确认后调 `CONFIRM_EXECUTE`。
5. 执行完成后刷新 `/runtime`，展示旧产物 + 新产物。

## 10. 注意事项

1. 完成态 `REPLAN` 不会切换到新的 taskId，前端不要因为重新规划而切换工作台任务。
2. 原地编辑 PPT / DOC 都不新增 artifact，判断是否成功看原 artifact 的 `status=UPDATED` 和 `preview`。
3. 整体重跑才会在执行完成后新增 artifact。
4. DOC 原地修改已接入，但并不是所有修改都会直接执行；高风险场景会先进入待确认计划。
5. 如果 `planningPhase=ASK_USER` 且 `actions.canConfirm=true`、`actions.canResume=true`，优先按“文档修改待确认态”处理，而不是普通澄清问题。
6. 如果 `planningPhase=ASK_USER` 且只有 `actions.canResume=true`，通常表示产物选择或补充澄清。
7. 如果任务处于 `EXECUTING`，不建议直接做完成态产物修改；应等待完成，或先取消/中断后再重新提交调整。
