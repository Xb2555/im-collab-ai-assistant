# 完成态任务产物修改接口说明

本文面向前端说明 GUI 侧如何对已完成任务继续调整，包括原地修改已有 PPT 产物，以及保留旧产物后重新规划并重跑任务。

## 1. 功能边界

已完成任务现在可以继续发起 `REPLAN`。

前端统一调用：

```http
POST /api/planner/tasks/{taskId}/commands
```

后端会根据用户反馈和产物策略分流：

> 重要限制：当前只支持 PPT 产物原地修改。DOC 产物原地修改能力暂未接入，前端不要展示为“可直接修改文档”。如果用户要调整 DOC，目前只能引导用户保留现有文档并重新生成一版，或提示等待 Doc 原地编辑能力合入。

| 场景 | 后端行为 | 是否新 taskId | 是否新增产物 |
| --- | --- | --- | --- |
| 明确修改已有 PPT，如“把第 1 页标题改成 xxx” | 原地编辑指定 PPT | 否 | 否 |
| 保留旧版、另做一版、整体重新规划、重新生成 | 重新进入 Planner，生成新计划 | 否 | 执行后新增产物 |
| 已完成任务只有一个可编辑 PPT，且用户明确修改内容 | 默认修改该 PPT | 否 | 否 |
| 已完成任务有多个可编辑产物但未指定产物 | 返回追问，让用户选择产物 | 否 | 否 |
| 修改 DOC | 当前 Doc 原地编辑暂未接入，会返回提示，不会直接修改旧文档 | 否 | 否 |

> 注意：`REPLAN` 只负责调整计划或触发原地修改。整体重新规划后通常会回到 `PLAN_READY`，前端仍需再调用 `CONFIRM_EXECUTE` 才会真正重跑。

## 2. 前端判断入口

查询运行态：

```http
GET /api/planner/tasks/{taskId}/runtime
```

当返回：

```json
{
  "data": {
    "task": {
      "status": "COMPLETED"
    },
    "actions": {
      "canReplan": true
    }
  }
}
```

说明该完成态任务允许继续调整。

## 3. 请求字段

接口：

```http
POST /api/planner/tasks/{taskId}/commands
Content-Type: application/json
```

请求体：

```json
{
  "action": "REPLAN",
  "feedback": "把第1页标题改成采购风险与建议",
  "artifactPolicy": "EDIT_EXISTING",
  "targetArtifactId": "artifact-id",
  "version": 3
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `action` | string | 是 | 固定传 `REPLAN`。 |
| `feedback` | string | 是 | 用户的调整说明。建议写完整自然语言，不要只传枚举。 |
| `artifactPolicy` | string | 否 | 产物策略。旧前端不传时按 `AUTO` 处理。 |
| `targetArtifactId` | string | 否 | 指定要修改的已有产物 ID。多产物场景建议传。 |
| `version` | number | 是 | 当前任务版本号，用于并发冲突校验。可从 `/runtime.data.task.version` 或 `GET /{taskId}` 获取。 |

`artifactPolicy` 可选值：

| 值 | 含义 | 推荐使用场景 |
| --- | --- | --- |
| `AUTO` | 让后端根据用户话术自动判断 | 旧逻辑兼容或不确定场景 |
| `EDIT_EXISTING` | 倾向修改已有产物 | 用户明确“修改这个 PPT/这一页” |
| `CREATE_NEW` | 倾向创建新产物 | 用户明确要新的产物形式 |
| `KEEP_EXISTING_CREATE_NEW` | 保留旧产物并新建一版 | 用户明确“保留旧版/另做一版/重新生成” |

建议前端同时传 `artifactPolicy` 和明确的 `feedback`。例如不要只传：

```json
{
  "artifactPolicy": "KEEP_EXISTING_CREATE_NEW"
}
```

推荐：

```json
{
  "artifactPolicy": "KEEP_EXISTING_CREATE_NEW",
  "feedback": "保留现有 PPT，整体重新规划并生成一份 3 页新版 PPT，主题为采购评审复盘"
}
```

## 4. 原地修改已有 PPT

适用条件：

1. 任务状态为 `COMPLETED`。
2. 任务已有 PPT artifact。
3. 用户明确说明要改哪一页、改成什么内容。
4. 如果存在多个 PPT，前端最好传 `targetArtifactId`。

请求示例：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "REPLAN",
    "feedback": "把第1页标题改成采购风险与建议",
    "artifactPolicy": "EDIT_EXISTING",
    "targetArtifactId": "59e8550f-12ee-4089-ab3e-a231983c32d4",
    "version": 3
  }'
```

典型响应：

```json
{
  "code": 0,
  "data": {
    "taskId": "task-id",
    "version": 3,
    "planningPhase": "COMPLETED",
    "assistantReply": "已修改 PPT 第 1 页：采购风险与建议",
    "actions": {
      "canConfirm": false,
      "canReplan": true,
      "canCancel": false,
      "canResume": false,
      "canInterrupt": false,
      "canRetry": false
    }
  },
  "message": "ok"
}
```

修改完成后，前端应再查 `/runtime` 刷新产物状态：

```http
GET /api/planner/tasks/{taskId}/runtime
```

产物会保持同一个 `artifactId` 和 `url`，不会新增同类型产物：

```json
{
  "data": {
    "task": {
      "status": "COMPLETED"
    },
    "artifacts": [
      {
        "artifactId": "59e8550f-12ee-4089-ab3e-a231983c32d4",
        "type": "PPT",
        "title": "原 PPT 标题",
        "url": "https://xxx.feishu.cn/slides/xxx",
        "preview": "已修改 PPT 第 1 页：采购风险与建议",
        "status": "UPDATED"
      }
    ]
  }
}
```

## 5. 保留旧产物并重新规划重跑

适用条件：

1. 用户明确说“保留旧版”“另做一版”“重新生成”“整体重新规划”“重跑”。
2. 最终希望新增产物，而不是覆盖现有 PPT。

请求示例：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "REPLAN",
    "feedback": "保留现有 PPT，整体重新规划并生成一份 3 页新版 PPT，主题采购评审复盘，三页分别是背景、风险、建议",
    "artifactPolicy": "KEEP_EXISTING_CREATE_NEW",
    "version": 3
  }'
```

典型响应：

```json
{
  "code": 0,
  "data": {
    "taskId": "task-id",
    "planningPhase": "PLAN_READY",
    "actions": {
      "canConfirm": true,
      "canReplan": true,
      "canCancel": true,
      "canResume": false,
      "canInterrupt": false,
      "canRetry": false
    }
  },
  "message": "ok"
}
```

前端此时应展示更新后的计划，并等待用户确认。用户确认后调用：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "CONFIRM_EXECUTE",
    "version": 4
  }'
```

执行完成后再查 `/runtime`，应看到旧 PPT 仍在，并新增一个新的 PPT artifact。

## 6. 多产物选择

如果同一个完成态任务下有多个可修改产物，且前端没有传 `targetArtifactId`，后端会进入追问态。

典型响应：

```json
{
  "code": 0,
  "data": {
    "planningPhase": "ASK_USER",
    "assistantReply": "这个任务下有多个可修改产物，你想修改哪一个？\n1. [PPT] 采购评审初版 965fcaac\n2. [PPT] 采购评审老板版 e5e34273\n回复编号即可。",
    "actions": {
      "canResume": true
    }
  },
  "message": "ok"
}
```

GUI 侧有两种处理方式：

1. 推荐：前端自己展示 artifacts，让用户选择后重新发 `REPLAN`，并带上 `targetArtifactId`。
2. 兼容：直接调用 `RESUME`，把用户选择的编号传给后端。

`RESUME` 示例：

```bash
curl -X POST 'http://localhost:8078/api/planner/tasks/{taskId}/commands' \
  -H 'Content-Type: application/json' \
  -d '{
    "action": "RESUME",
    "feedback": "2",
    "version": 4
  }'
```

如果编号不存在，后端会继续返回提示，不会误改产物。

## 7. DOC 产物修改限制

当前版本对 DOC 的支持边界如下：

1. **不支持 DOC 原地修改**：用户说“修改这份文档”“给文档补一段”“润色已有文档”时，后端不会直接编辑旧 DOC。
2. **不会伪装成修改成功**：如果命中 DOC 原地修改意图，后端会返回提示，说明 Doc 原地编辑能力暂未接入。
3. **可走重新生成**：如果用户明确说“保留现有文档，再生成一版”“重新生成新版文档”，可以走完成态 `REPLAN`，后续确认执行后新增 DOC 产物。
4. **前端展示建议**：artifact type 为 `DOC` 时，不要展示“直接修改原文档”按钮；可以展示“保留旧版并重新生成”。

DOC 修改请求示例：

```json
{
  "action": "REPLAN",
  "feedback": "把这份文档补充一段风险分析",
  "artifactPolicy": "EDIT_EXISTING",
  "targetArtifactId": "doc-artifact-id",
  "version": 3
}
```

典型响应会进入追问/提示态：

```json
{
  "code": 0,
  "data": {
    "planningPhase": "ASK_USER",
    "assistantReply": "Doc 原地编辑能力暂未接入。你是要保留现有文档再新建一版，还是等 Doc 原地编辑能力合入后再直接修改旧文档？",
    "actions": {
      "canResume": true
    }
  },
  "message": "ok"
}
```

如果用户选择保留旧文档并重新生成，前端应改为发送：

```json
{
  "action": "REPLAN",
  "feedback": "保留现有文档，重新生成一份新版文档，并补充风险分析章节",
  "artifactPolicy": "KEEP_EXISTING_CREATE_NEW",
  "version": 4
}
```

## 8. 关键响应字段

### `PlanPreviewVO`

`POST /commands` 返回 `PlanPreviewVO`：

| 字段 | 说明 |
| --- | --- |
| `taskId` | 当前任务 ID。完成态调整不会生成新 taskId。 |
| `version` | 当前任务版本。后续命令需要带最新版本。 |
| `planningPhase` | 当前阶段，如 `PLAN_READY`、`ASK_USER`、`EXECUTING`、`COMPLETED`。 |
| `cards` | 当前计划卡片。重新规划后会更新。 |
| `actions` | 前端可执行动作。 |
| `assistantReply` | 后端给用户看的提示，如“已修改 PPT 第 1 页”。 |
| `accepted` | 是否被 Agent 接收为真实任务。 |
| `runtimeAvailable` | 是否可以查 `/runtime`。 |
| `transientReply` | 是否只是提示型回复，不应进入工作台。 |
| `capabilityHints` | 当前计划相关能力提示。 |

### `TaskDetailVO`

`GET /runtime` 返回 `TaskDetailVO`：

| 字段 | 说明 |
| --- | --- |
| `task` | 任务摘要，含 `status/version/progress`。 |
| `steps` | 当前步骤列表。 |
| `artifacts` | 产物列表，前端主要从这里拿 `artifactId/url/status/preview`。 |
| `events` | 时间线事件。 |
| `actions` | 当前运行态可执行动作。 |

### `TaskArtifactVO`

| 字段 | 说明 |
| --- | --- |
| `artifactId` | 产物 ID。原地编辑时保持不变。 |
| `type` | 产物类型，如 `PPT`、`DOC`。 |
| `title` | 产物标题。 |
| `url` | 飞书产物链接。原地编辑时保持不变。 |
| `preview` | 最近一次修改摘要。 |
| `status` | `CREATED`、`UPDATED` 等。 |
| `createdAt` | 创建时间。 |

## 9. 前端推荐交互

完成态任务详情页：

1. 先查 `/runtime`。
2. 如果 `actions.canReplan=true`，展示“调整任务/修改产物”入口。
3. 如果用户选择“修改已有产物”，前端展示 artifact 列表，要求用户选择具体产物。
4. 对 PPT 修改，要求用户输入明确修改内容，如“第几页”“改成什么”。
5. 如果 artifact type 是 `PPT`，调 `REPLAN + EDIT_EXISTING + targetArtifactId`。
6. 如果 artifact type 是 `DOC`，不要走原地修改，提示当前暂不支持，或引导“保留旧版并重新生成”。
7. 响应成功后立刻刷新 `/runtime`，展示同一个链接和 `preview`。

完成态整体重跑：

1. 用户选择“保留旧版并重新生成”。
2. 调 `REPLAN + KEEP_EXISTING_CREATE_NEW`，`feedback` 中写明新产物要求。
3. 若返回 `PLAN_READY`，展示新计划。
4. 用户确认后调 `CONFIRM_EXECUTE`。
5. 执行完成后刷新 `/runtime`，展示旧产物 + 新产物。

## 10. 注意事项

1. 完成态 `REPLAN` 不会切换到新的 taskId，前端不要因为重新规划而切换工作台任务。
2. 原地编辑 PPT 不新增 artifact，判断是否成功看原 artifact 的 `status=UPDATED` 和 `preview`。
3. 整体重跑才会在执行完成后新增 artifact。
4. 修改 DOC 当前还没有接入原地编辑能力，前端不要把 DOC 修改伪装成已支持，也不要展示“直接修改原文档”作为可用能力。
5. 如果 `planningPhase=ASK_USER` 且 `actions.canResume=true`，说明后端需要用户补充选择或澄清。
6. 如果任务处于 `EXECUTING`，不建议直接做完成态产物修改；应等待完成，或先取消/中断后再重新提交调整。
