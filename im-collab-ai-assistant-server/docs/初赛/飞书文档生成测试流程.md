# 飞书文档生成测试流程

## 1. 目的

本文面向场景 B `planner` 与场景 C `harness` 的联调测试，目标是按真实全链路顺序验证以下能力：

1. 自然语言需求能被正确理解并生成文档型计划。
2. 澄清问题能按需出现，并在补充回答后继续规划。
3. 用户确认执行后，任务能进入 harness 文档工作流。
4. 文档生成流程能产出飞书文档链接。
5. 运行时状态、事件流、artifact 和任务卡片状态能正确投影。
6. 对于 review 阶段无法自动解决的问题，系统会进入 `WAITING_APPROVAL`。

## 2. 预置条件

开始测试前请确认：

1. 服务已启动。
2. LLM 配置可用。
3. Redis 可用。
4. `lark-cli` 已登录且具备文档创建权限。
5. 当前文档生成链路已经接入：
   - planner
   - harness
   - skills.lark.doc

本文示例统一使用：

- 服务地址：`http://localhost:8123`
- 目标接口前缀：`/planner/tasks`

如果你的服务端口不同，请自行替换。

## 3. 推荐测试用例

建议优先测试以下这个用例，因为它同时覆盖：

1. 文档型任务
2. 技术架构模板
3. Mermaid 图需求
4. review 自动补齐
5. 飞书文档写入

测试输入：

`帮我写一篇 harness 架构设计文档，面向开发者汇报，需要包含 Mermaid 全局架构流程图和数据流转图`

## 4. 全流程测试步骤

## 一、创建任务规划

### 1.1 调用接口

`POST /planner/tasks/plan`

### 1.2 请求参数

```json
{
  "rawInstruction": "帮我写一篇 harness 架构设计文档，面向开发者汇报，需要包含 Mermaid 全局架构流程图和数据流转图",
  "workspaceContext": {
    "selectionType": "MESSAGE",
    "selectedMessages": [
      "当前项目已经完成场景A和场景B，准备推进场景C harness。",
      "文档主要面向开发者，需要说明 planner、harness、skills、store 的关系。"
    ],
    "profession": "后端开发工程师",
    "audience": "开发团队",
    "tone": "正式、技术化",
    "language": "中文",
    "promptProfile": "default",
    "promptVersion": "v4"
  }
}
```

### 1.3 curl 示例

```bash
curl -X POST 'http://localhost:8123/planner/tasks/plan' \
  -H 'Content-Type: application/json' \
  -d '{
    "rawInstruction": "帮我写一篇 harness 架构设计文档，面向开发者汇报，需要包含 Mermaid 全局架构流程图和数据流转图",
    "workspaceContext": {
      "selectionType": "MESSAGE",
      "selectedMessages": [
        "当前项目已经完成场景A和场景B，准备推进场景C harness。",
        "文档主要面向开发者，需要说明 planner、harness、skills、store 的关系。"
      ],
      "profession": "后端开发工程师",
      "audience": "开发团队",
      "tone": "正式、技术化",
      "language": "中文",
      "promptProfile": "default",
      "promptVersion": "v4"
    }
  }'
```

### 1.4 期望结果

理想情况下：

1. `planningPhase = PLAN_READY`
2. `title` 应接近 “harness 架构设计文档” 而不是被改写成别的主题
3. `cards` 里至少有 1 张 `DOC` 卡片
4. 不应出现 `PPT` 卡片
5. `clarificationQuestions` 应为空或极少

如果信息仍不足，也允许出现：

1. `planningPhase = ASK_USER`
2. 返回 1-3 个问题
3. 问题应围绕真正缺失的信息，而不是固定三问

## 二、如进入 ASK_USER，补充澄清信息

### 2.1 触发条件

当步骤一返回：

`planningPhase = ASK_USER`

时执行本步骤；如果步骤一已直接 `PLAN_READY`，跳过本步骤。

### 2.2 调用接口

`POST /planner/tasks/{taskId}/resume`

### 2.3 请求参数

```json
{
  "feedback": "面向开发者汇报，需要解释 planner、harness、skills、store 的职责边界，并且必须包含 Mermaid 全局架构图和数据流转图",
  "replanFromRoot": false
}
```

### 2.4 curl 示例

```bash
curl -X POST 'http://localhost:8123/planner/tasks/{taskId}/resume' \
  -H 'Content-Type: application/json' \
  -d '{
    "feedback": "面向开发者汇报，需要解释 planner、harness、skills、store 的职责边界，并且必须包含 Mermaid 全局架构图和数据流转图",
    "replanFromRoot": false
  }'
```

### 2.5 期望结果

1. `planningPhase = PLAN_READY`
2. `clarificationAnswers` 已写入刚刚的反馈
3. `cards` 中仍然只有 `DOC` 卡片
4. 计划摘要中体现“面向开发者”“Mermaid 图”等约束

    "version": 3
  }'

### 5.4 期望结果

1. `planningPhase = EXECUTING`
2. planner 内部调用 harness 执行入口
3. 文档工作流开始运行：
   - `dispatch_doc_task`
   - `generate_outline`
   - `generate_sections`
   - `plan_diagrams`
   - `generate_mermaid`
   - `validate_mermaid`

```bash

```

### 7.3 期望结果

1. `task.status`
   - 执行中应为 `EXECUTING`

   - 应出现文档大纲、Mermaid 图源码、最终文档链接等 artifact
4. `actions`

## 十、人工确认分支测试

### 10.1 触发目标

故意制造一个 review 无法自动解决的问题，例如：

`帮我写一篇 harness 架构设计文档，但必须完全基于我即将补充的一份内部规范来写。`

同时不提供那份规范。

### 10.2 调用步骤

仍按前面步骤：

1. `POST /planner/tasks/plan`
2. 如有需要 `POST /planner/tasks/{taskId}/resume`
3. `POST /planner/tasks/{taskId}/commands` with `CONFIRM_EXECUTE`
4. `GET /planner/tasks/{taskId}/runtime`

### 10.3 期望结果

如果系统无法通过 `supplementalSections` 自动补齐，应该出现：

1. task 或 step 状态进入 `WAITING_APPROVAL`
2. 事件中出现 `APPROVAL_REQUESTED`
3. `runtime` 快照中能看到等待人工确认的迹象

### 10.4 当前限制

当前代码里已经存在内部的 harness 恢复能力：

- `HarnessFacade.resumeExecution(taskId, approval)`
- `ExecutionOrchestrator.resume(taskId, approval)`

但目前对外 HTTP 层没有暴露单独的 approval 提交接口。

这意味着：

1. 你可以测到系统进入 `WAITING_APPROVAL`
2. 也可以验证事件流和 runtime 快照
3. 但不能仅通过现有公开 HTTP 接口把 `Approval` 再提交回 harness 继续执行

这属于当前测试边界，后续建议补一个对外 approval API。

## 十二、关键通过标准

本轮飞书文档生成测试可判为通过，至少满足：

1. 用户明确要求文档时，不出现多余 PPT 卡片。
2. 计划标题和执行标题不偏离原始需求。
3. 文档可成功写入飞书并返回链接。
4. 文档中不出现“待补充”或空章节。
5. 文档标题层级和编号规范符合要求。
6. Mermaid 图能按约定风格生成并写入。
7. review 可自动补齐缺项并二次复审。
8. 对于无法自动解决的问题，系统能进入 `WAITING_APPROVAL`。

## 5. “第二次 review 只保留仍然无法自动解决的问题”是什么意思

这句话的含义是：

1. 第一轮 `review` 先检查当前文档是否缺章节、缺结论、缺图、缺关键说明。
2. 如果 `review` 自己已经给出了 `supplementalSections`，系统会先自动把这些补丁合并到正文里。
3. 合并完成后，系统会再跑一次 `review`。
4. 第二次 `review` 的职责不是重复报第一次的问题，而是只保留“补丁合并之后仍然存在的问题”。

例如：

1. 第一轮发现“缺少设计目标与非目标”。
2. review 同时生成了这个章节的补充内容。
3. 系统先自动合并。
4. 第二轮 review 如果确认这个问题已经解决，就不再要求人工确认。

只有下面这类问题才应该进入人工确认：

1. 缺少外部事实，系统无法自行臆断
2. 缺少必须由用户拍板的业务判断
3. 用户要求的专有资料根本没有提供
4. 自动补丁后仍然和原始目标冲突

## 6. 这是不是会触发一次中断，让用户确认

可以理解为“进入等待人工确认的暂停态”，但不是 planner 那种重新澄清的 `ASK_USER`，也不是简单的系统崩溃中断。

更准确地说，是：

`场景 C 执行过程中的人工审批暂停`

它的含义是：

1. planner 已经完成并进入执行
2. harness 在执行文档生成时遇到无法自动闭环的问题
3. 系统发出 `APPROVAL_REQUESTED`
4. 任务状态切到 `WAITING_APPROVAL`
5. 等待用户给出批准、修改意见或拒绝

## 7. 用户确认什么

确认的不是“要不要开始执行”，那个动作在 `CONFIRM_EXECUTE` 阶段已经完成了。

这里确认的是：

1. 当前文档是否可以接受现状继续推进
2. 是否需要补充某些缺失资料
3. 是否接受系统建议的补齐方向
4. 是否要改写目标再继续生成
5. 是否直接拒绝并终止本次执行

在代码语义上，对应 `ApprovalStatus`：

1. `APPROVED`
2. `MODIFIED`
3. `REJECTED`

## 8. 确认之后会发生什么

当前内部实现是这样的：

1. `APPROVED`
   - harness 从当前 step 继续执行
2. `MODIFIED`
   - 把 `userFeedback` 写回任务，再从当前 step 继续执行
3. `REJECTED`
   - 任务直接终止，状态进入 `ABORTED`

也就是说，人工确认之后不是重新走场景 B 规划，而是继续恢复场景 C 的执行链路。

但要注意当前现状：

1. 内部 Java 层已经支持这套逻辑
2. 公开 HTTP 层暂时还没有暴露 approval 提交接口

所以从“产品设计”上讲，这个流程已经成立；从“当前可直接用 curl 完整压测”上讲，还差一个对外 approval API。
