# 0504 CLI 更新后的架构调整

## 1. 结论

当前 `document-iteration` 已经从纯 patch 链路，演化成“意图识别 + 结构解析 + 计划编排 + 富媒体执行”的混合架构，但没有完全收口。

CLI 升级 / 同事把部分能力改走 HTTP 后，现状是：

1. 文本类文档迭代已经基本可用。
2. 富媒体链路已经接入，但仍有重 fetch、重校验、资产输入不足等问题。
3. `LarkDocTool` 同时承担了 CLI、OpenAPI、Markdown 转换三类职责，边界已经不清晰。

---

## 2. 当前已完成

### 2.1 文本迭代主链路

1. `DocumentEditIntentResolver` 已能输出高层意图与槽位。
2. `DocumentStructureSnapshotBuilder` 已能构建结构快照。
3. `DocumentAnchorResolver` 已能基于快照做定位。
4. `DocumentPatchCompiler` 已能把文本编辑编译成 patch 计划。
5. `DocumentPatchExecutor` 已能执行文本 patch。
6. `DocumentTargetStateVerifier` 已承担最终结构校验职责。

### 2.2 富媒体规划

1. `INSERT_MEDIA` 已进入专门分支。
2. `AssetResolutionFacade` 已支持图片 / 表格 / 白板资产描述。
3. `RichContentExecutionPlanner` 已生成富媒体执行步骤。
4. `RichContentExecutionEngine` 已接入执行器框架。

### 2.3 文档创建

1. `createDoc` 已改为走 OpenAPI。
2. Markdown 创建不再依赖旧式 CLI 生成整篇文档。

---

## 3. 已修复但未闭环

### 3.1 兼容层已存在

当前代码已经能同时处理：

1. CLI 文档读写。
2. OpenAPI 文档创建。
3. Markdown 转 block。

但这些能力是并列存在，不是清晰分层。

### 3.2 富媒体执行已接入，但仍不稳定

当前问题不是“识别不了图片”，而是：

1. 资产来源可能为空。
2. `upload_image` 仍然依赖 CLI 语义。
3. `fetchDocFull(..., with-ids)` 太重，容易在读取阶段超时。

---

## 4. 仍然欠缺

### 4.1 CLI 与 OpenAPI 边界未收口

当前 `LarkDocTool` 同时承担：

1. CLI 命令组装。
2. CLI 兼容探测。
3. OpenAPI 创建。
4. Markdown block 转换。
5. 富媒体命令 fallback。

这会导致：

1. 超时难定位。
2. 升级后兼容逻辑难维护。
3. 一处变更会影响全链路。

### 4.2 富媒体链路仍缺三个闭环

1. 图片资产闭环。
2. 轻量快照闭环。
3. 最终状态校验闭环。

### 4.3 `DocumentStructureSnapshotBuilder` 仍偏重

它现在仍会尝试读取全文 XML 和全文 Markdown。
对大文档，这会成为富媒体流程的瓶颈。

---

## 5. mermaid 变成代码块的判断

从当前代码和 git 记录看，问题大概率出在 `289f3d2 fix: render lark doc markdown blocks` 这一轮。

推断链路是：

1. 文档内容先按 Markdown 转 block。
2. `LarkDocTool` 的渲染/转换逻辑把 fenced mermaid 当成普通代码块处理。
3. 飞书文档最终收到的是代码块节点，不是 mermaid 图节点。

这不是单一 bug，更像是“Markdown 文本渲染”覆盖了“富节点语义”。

---

## 6. 推荐架构

### 6.1 读写分层

建议把 `LarkDocTool` 拆成三层：

1. `LarkDocCliAdapter`：只负责 CLI 命令和兼容。
2. `LarkDocOpenApiAdapter`：只负责 OpenAPI 创建/上传/轻量读。
3. `LarkDocContentCodec`：只负责 Markdown / block / 富节点编码转换。

### 6.2 迭代链路分层

建议 `document-iteration` 明确分成：

1. 意图识别。
2. 结构快照。
3. 锚点解析。
4. 计划编排。
5. CLI/OpenAPI 执行。
6. 结构校验。

### 6.3 富媒体策略

1. 图片插入优先使用附件 / URL / 生成描述，缺一则澄清。
2. 白板 / 表格 / 图片不要再走统一文本 patch。
3. 全量 `with-ids` 抓取只保留给必要的结构校验，不放在主执行路径。

---

## 7. 当前建议的工程取舍

1. 先保留 CLI 作为兼容执行层。
2. 逐步把创建、上传、轻量查询迁到 HTTP。
3. 富媒体只保留“执行计划 + 资产输入 + 轻量校验”。
4. 逐步删掉靠文本猜语义的 fallback。

---

## 8. 结论

当前代码不是“没做完”，而是“做到了半层混合架构”。

下一步不是继续补更多 CLI 兼容，而是把：

1. 文本 patch。
2. 富媒体执行。
3. OpenAPI 创建。
4. Markdown 渲染。

四条线彻底分开。
