## 3. 测试结论

### 3.1 已达到的能力

当前链路已经做到以下几点：

1. 能识别用户意图为 `INSERT_MEDIA`。
2. 能进一步识别语义动作 `INSERT_IMAGE_AFTER_ANCHOR`。
3. 能识别这类操作属于高风险操作并进入审批流。
4. 服务层已经接入富媒体规划对象：
   - `AssetResolutionFacade`
   - `RichContentExecutionPlanner`
   - `RichContentTargetStateVerifier`

这说明当前代码已经不再是完全没有富媒体能力，而是已经完成了“富媒体规划态”的一部分重构。

### 3.2 未达到的能力

当前版本仍然不具备“真正可用的文档插图能力”，原因如下：

1. 审批通过后没有真正执行富媒体插入。
2. 返回结果中：
   - `toolCommandType = null`
   - `modifiedBlocks = []`
   - `generatedContent = ""`
   说明没有真正生成可执行的文档写入动作。
3. `targetTitle` 和 `targetPreview` 都为空，说明锚点解析没有稳定落到“项目背景”。
4. 接口返回 `planningPhase = COMPLETED`，但实际文档没有变化，这会误导调用方。

结论：

**当前实现只完成了“富媒体编辑计划生成与审批流接入”，没有完成“审批后真正执行富媒体插入”。**

## 6. 最佳修改方案

要求：

1. 不能靠中文关键词硬编码。
2. 不能继续把富媒体能力塞进 `patchCompiler`。
3. 不能用全文 markdown 拼接模拟图片、表格、白板插入。
4. 必须沿用当前 `document-iteration` 新架构的结构化分层思路。

### 6.1 把富媒体操作正式提升为“执行计划驱动”

当前问题的根因不是“识别不到图片”，而是：

**系统只生成了 `executionPlan`，却没有真正执行它。**

必须新增独立执行器，例如：

- `RichContentExecutionEngine`

职责：

1. 读取 `DocumentEditPlan.executionPlan`
2. 顺序执行每个 `ExecutionStep`
3. 对每一步产出中间结果
4. 在失败时按 `rollbackPolicy` 中止或回滚

这层不能继续复用 `DocumentPatchExecutor`，因为富媒体不是纯 patch。

### 6.2 富媒体执行链必须分为四段

推荐固定为以下四段：

1. `Asset Resolve`
   - 从自然语言或上下文中得到结构化 `MediaAssetSpec`
   - 图片、表格、白板分别建模

2. `Asset Materialize`
   - 图片：上传或生成文件 token
   - 表格：生成 schema 和 rows
   - 白板：生成 DSL 或创建白板资源

3. `Document Node Insert`
   - 在正确锚点后插入媒体节点
   - 返回新 block id

4. `Target State Verify`
   - 校验新节点是否真实存在
   - 校验插入位置是否正确
   - 校验节点类型是否符合预期

### 6.3 锚点解析必须回到结构快照，不要靠词面兜底

当前测试里：

- `targetTitle = ""`
- `targetPreview = ""`

说明“项目背景后”没有正确解析到目标结构节点。

这部分必须改成：

1. 基于 `DocumentStructureSnapshot`
2. 基于 heading/block/tree 结构
3. 让 LLM 只返回结构化定位槽位
4. 由本地 resolver 在 snapshot 上确定唯一锚点

不要做：

1. `if (instruction.contains("项目背景"))`
2. `if (instruction.contains("后"))`
3. `if (instruction.contains("图片"))`

正确做法应是：

1. LLM 输出：
   - `targetRegion`
   - `targetSemantic`
   - `targetKeywords`
   - `assetType`
2. 本地 resolver 再把这些槽位映射到唯一 block/section

### 6.4 `DocumentEditIntentResolver` 必须真正产出 `assetSpec`

当前代码虽然有 `assetSpec` 字段，但链路没有稳定填充。

必须让 `DocumentEditIntentResolver` 或独立的 `AssetSpecResolver` 输出结构化资产规格：

图片至少应包含：

1. `assetType = IMAGE`
2. `sourceType`
3. `sourceRef`
4. `caption`
5. `altText`
6. `generationPrompt`

表格至少应包含：

1. `assetType = TABLE`
2. `tableSchema`
3. `generationPrompt`

白板至少应包含：

1. `assetType = WHITEBOARD`
2. `whiteboardDsl`
3. `sourceRef`

这样 `AssetResolutionFacade` 才有真实输入，而不是空转。

### 6.5 审批通过后的执行分流必须修改

当前审批通过后统一走：

- `patchExecutor.execute(...)`

这不适用于富媒体。

必须改成：

1. 文本类编辑：
   - 继续走 `patchExecutor`

2. 富媒体/布局类编辑：
   - 改走 `RichContentExecutionEngine`

然后再根据类型分别走：

1. `targetStateVerifier`
2. `richContentTargetStateVerifier`

这一步是本次问题的核心修复点。

### 6.6 `COMPLETED` 状态语义必须真实

当前审批通过后虽然返回：

- `planningPhase = COMPLETED`

但文档没有实际变化。

这会误导前端和调用方。

必须保证：

1. 只有当富媒体节点真实落盘后，才能返回 `COMPLETED`
2. 如果只是审批通过但待执行，应有独立中间态
3. 如果执行计划为空或未执行，不允许伪装成完成

### 6.7 不要把富媒体插入退化成文本 patch

禁止以下错误修法：

1. 用 `append markdown image` 伪装图片插入
2. 用 `str_replace` 插入图片链接
3. 用 `block_replace` 拼接图片占位文字
4. 用“插入一段说明文本”代替真实白板/表格/图片节点

这类改法只会把富媒体重新做成文本假象，后续一定再次失控。

## 8. 预期交互过程

当前项目代码和设计方向决定了，富媒体插入的正确交互不应该是只发一句：

- `在项目背景后插入一张图片`

而应该是：

1. 先确定目标文档。
2. 再提供富媒体资产来源。
3. 再指定插入位置。
4. 再发起文档迭代请求。
5. 系统生成待审批计划。
6. 用户确认后，系统执行富媒体插入。

### 8.1 图片插入的推荐交互

图片来源应支持三类：

1. 已上传附件
2. 外部图片链接
3. AI 生成图片

对应交互建议如下。

#### 方式 A：先上传附件，再插入

用户动作：

1. 在聊天或前端界面上传图片。
2. 选中文档。
3. 发送指令：
   - `把这张图片插入到项目背景后`

系统行为：

1. 前端把附件引用写入 `workspaceContext.attachmentRefs`
2. 后端从附件引用中解析资产
3. 后端再结合自然语言解析插入位置

#### 方式 B：直接提供图片链接

用户动作：

1. 选中文档。
2. 发送指令：
   - `把这张图片 https://example.com/a.png 插入到项目背景后`

系统行为：

1. 后端从指令或结构化字段中提取 `sourceRef`
2. 将其解析为图片资产
3. 再执行插入计划

#### 方式 C：AI 生成图片后插入

用户动作：

1. 选中文档。
2. 发送指令：
   - `生成一张系统架构图，并插入到项目背景后`

系统行为：

1. 后端将生成描述写入 `generationPrompt`
2. 先走图片生成或图片创建流程
   * mermaid 架构图走代码创建，如果是其他图片走第三方api生图接口

3. 再将生成结果插入文档

### 8.2 表格插入的推荐交互

用户动作：

1. 选中文档。
2. 发送指令：
   - `在项目目标后插入一个里程碑表格，包含阶段、负责人、截止时间三列`

系统行为：

1. 后端解析出 `assetType = TABLE`
2. 生成结构化 `tableSchema / tableData`
3. 再执行表格节点插入

### 8.3 白板插入的推荐交互

用户动作：

1. 选中文档。
2. 发送指令：
   - `在方案概述后插入一个白板流程图`

系统行为：

1. 后端解析出 `assetType = WHITEBOARD`
2. 生成 whiteboard DSL 或白板创建参数
3. 创建白板实体
4. 在文档中插入白板引用节点

### 8.4 当前阶段不应承诺的交互

在当前代码状态下，不应把下面这些说法当成“已经正式支持”：

1. 只发一句 `插入一张图片` 就能成功插入真实图片
2. 只说 `插入视频链接` 就能成功插入视频
3. 不提供附件、链接、生成描述，也能让系统自动知道是哪张图片

原因是当前链路还没有稳定完成：

1. `assetSpec` 解析
2. 富媒体执行计划真正执行
3. 审批后的真实落盘闭环
