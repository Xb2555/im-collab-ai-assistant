## 2. 当前真实状态判断

先给出当前状态的工程判断，避免后续误改。

### 2.1 已经到位的部分

当前代码已经具备以下基础：

1. `DocumentEditIntent` 已支持 `assetSpec`
2. `DocumentEditPlan` 已支持：
   - `resolvedAssetSpec`
   - `executionPlan`
3. 服务层已接入：
   - `AssetResolutionFacade`
   - `RichContentExecutionPlanner`
   - `RichContentTargetStateVerifier`
4. 富媒体语义已经进入 `DocumentSemanticActionType`
5. `DocumentPatchCompiler` 已经把富媒体/布局类动作从文本 patch 主路径中隔离出来，改为 `approval-only plan`

说明：

1. 当前代码不是完全没有富媒体架构。
2. 当前问题不是“识别不了图片”，而是“识别出来了，但没有执行闭环”。

### 2.2 当前真正的断点

结合测试结果和代码，当前断点有四个：

1. **服务层仍然只会执行 `patchExecutor`**
2. **`executionPlan.steps` 只是模型对象，没有执行器消费**
3. **锚点没有稳定解析到目标结构节点**
4. **`assetSpec -> resolvedAsset` 仍然输入不足，资产闭环不完整**

因此当前的真实状态应定义为：

**富媒体规划链路已接入，富媒体执行链路未闭环。**

## 3. 当前问题的代码级根因

### 3.1 `DefaultDocumentIterationExecutionService` 仍以 patch 为唯一执行机制

当前实现中：

1. `execute(...)` 会在富媒体动作时生成 `executionPlan`
2. 但 `applyAndComplete(...)` 仍统一调用：
   - `patchExecutor.execute(docRef, editPlan)`

这意味着：

1. 富媒体计划被生成了
2. 但真正执行时仍走文本 patch 执行器
3. 因为富媒体 plan 没有 `patchOperations`，所以不会写入任何内容

这就是测试里：

1. `toolCommandType = null`
2. `modifiedBlocks = []`
3. 飞书文档无变化

的直接根因。

### 3.2 `RichContentExecutionPlanner` 只有“步骤描述”，没有“步骤执行”

当前 `RichContentExecutionPlanner` 已能生成：

1. `UPLOAD_IMAGE`
2. `INSERT_IMAGE_BLOCK`
3. `VERIFY_IMAGE_NODE`
4. `INSERT_TABLE_BLOCK`
5. `WRITE_TABLE_DATA`
6. `CREATE_WHITEBOARD`
7. `INSERT_WHITEBOARD_REF`

但这些 `ExecutionStep` 只是数据结构，不会自动执行。

缺失的是：

1. `RichContentExecutionEngine`
2. `ExecutionStepHandlerRegistry`
3. 每类 step 的实际 handler

### 3.3 锚点解析仍然不稳定，导致目标位置为空

测试结果里：

1. `targetTitle = ""`
2. `targetPreview = ""`

这说明当前“在项目背景后插入一张图片”没有稳定解析到目标 block/section。

当前问题不一定是完全解析失败，而是：

1. `ResolvedDocumentAnchor` 的富媒体插入没有稳定转成可展示、可执行的 anchor
2. `toPlanVO()` 仍然主要依赖旧 `selector`
3. 富媒体 action 进入 VO 时没有把 section/block anchor 正确展示出来

### 3.4 `AssetResolutionFacade` 已存在，但前置 `assetSpec` 输入不稳定

当前 `AssetResolutionFacade` 的问题不是“完全没实现”，而是：

1. 输入依赖 `MediaAssetSpec`
2. 但当前请求链路缺少证据表明 `DocumentEditIntentResolver` 已稳定生成完整资产规格

特别是图片请求：

1. “插入一张图片”到底来自哪里
2. 是已有图片、外链图片、AI 生成，还是占位图

如果 `assetSpec` 不完整，`ResolvedAsset` 只能停留在空壳状态。

### 3.5 `RichContentTargetStateVerifier` 过早依赖“新 block id 已知”

当前 `RichContentTargetStateVerifier` 试图通过 `plan.getPatchOperations()` 推断新 block id。

这是不成立的，因为：

1. 富媒体链路现在本来就不走 patch
2. 新媒体 block id 应该来自执行器的运行时产物
3. 不是来自 patch operation

因此即使后续执行器接上，当前 verifier 也会失效。

### 3.6 当前 `DocumentStructureNode` 仍偏文本，不足以支撑稳定富媒体校验

当前 `DocumentStructureNode` 只有：

1. `blockId`
2. `blockType`
3. `headingLevel`
4. `titleText`
5. `plainText`
6. 父子/相邻关系

缺少：

1. `nodeType`
2. `mediaMeta`
3. `tableMeta`
4. `layoutMeta`
5. `rawPlatformPayload`

这会导致：

1. 图片节点只能用宽松 `blockType.contains("media")` 判断
2. 表格节点只能靠 `blockType.contains("table")`
3. 白板节点缺乏稳定引用信息

v1 可以先不一次性全补齐，但必须至少补“媒体引用层面的元数据”。

## 4. v1 的正确目标边界

v1 不是最终版多模态文档系统，而是**执行闭环版**。

v1 只要求把以下四条打通：

1. 图片插入闭环
2. 表格插入闭环
3. 白板引用插入闭环
4. 服务层执行分流闭环

v1 不强求一次解决：

1. 复杂布局系统
2. 图片替换/删除全量能力
3. 表格局部 cell 编辑
4. 白板复杂更新
5. 完整平台无关富节点抽象

原因：

1. 当前最急的不是“能力覆盖面”，而是“审批通过后根本不执行”。
2. 如果不先把 engine 闭环，继续加语义、加 planner、加 verifier 都是空转。

## 5. 用户端期待的正确交互流程

这部分优先级很高，因为后端方案必须服务于用户端真实体验，而不是只让接口“看起来能跑”。

### 5.1 总原则

用户端对“文档插入富媒体”的正确期待，不应该是：

1. 只说一句 `插入一张图片`
2. 系统就在没有图片来源、没有锚点确认、没有审批反馈的情况下神奇完成插入

用户端应该得到的是一条**可理解、可确认、可回看、可重试**的交互链路。

正确体验应满足：

1. 用户知道要插入到哪份文档
2. 用户知道插入到哪个位置
3. 用户知道插入的是什么资产
4. 用户知道系统是否还缺资产来源或描述
5. 用户知道是否需要审批
6. 审批通过后能看到真实执行过程和真实结果

### 5.2 图片插入的理想交互

以“在项目背景后插入一张图片”为例，用户端正确流程应该是：

1. 用户先选择文档，或在文档上下文中发起操作
2. 用户提供图片来源
3. 用户说明插入位置
4. 系统生成待确认计划
5. 用户确认
6. 系统执行插入
7. 系统返回真实结果

这条链路在产品上应该呈现为三种合法入口：

#### 方式 A：先上传图片，再插入

用户动作：

1. 上传图片
2. 选中文档
3. 发送指令：
   - `把这张图片插入到项目背景后`

系统反馈：

1. 识别目标文档
2. 识别附件图片
3. 识别插入位置“项目背景后”
4. 返回待审批计划：
   - 文档：某文档
   - 位置：`一、项目背景` 之后
   - 资产：已上传图片
   - 风险：高
5. 用户确认后执行插入
6. 返回：
   - 已插入成功
   - 新增 block id
   - 文档链接

#### 方式 B：提供图片链接后插入

用户动作：

1. 选中文档
2. 发送指令：
   - `把这张图片 https://example.com/a.png 插入到项目背景后`

系统反馈：

1. 识别图片链接
2. 解析插入位置
3. 生成待审批计划
4. 审批通过后执行

#### 方式 C：AI 生成图片再插入

用户动作：

1. 选中文档
2. 发送指令：
   - `生成一张系统架构图，并插入到项目背景后`

系统反馈：

1. 识别为“先生成资产，再插入文档”
2. 生成待审批计划：
   - 资产类型：图片
   - 来源：AI 生成
   - 生成描述：系统架构图
   - 插入位置：`一、项目背景` 之后
3. 用户确认
4. 系统先生成图片，再上传，再插入文档
5. 返回真实落盘结果

### 5.3 表格插入的理想交互

用户指令示例：

1. `在项目目标后插入一个里程碑表格，包含阶段、负责人、截止时间三列`

正确交互应该是：

1. 系统识别文档
2. 系统识别插入位置
3. 系统识别这是“生成表格后插入”
4. 系统给出待确认计划：
   - 插入位置
   - 表格结构摘要
   - 风险等级
5. 用户确认
6. 系统插入表格节点并写入数据
7. 返回真实结果

### 5.4 白板插入的理想交互

用户指令示例：

1. `在方案概述后插入一个白板流程图`

正确交互应该是：

1. 系统识别目标文档
2. 系统识别目标章节
3. 系统识别需要创建白板资源
4. 系统给出待确认计划：
   - 插入位置
   - 白板类型
   - 是否需要创建外部白板资源
5. 用户确认
6. 系统创建白板
7. 系统把白板引用插入文档
8. 返回真实结果

### 5.5 当前阶段必须补的交互缺口

结合当前测试结果，用户端现在最不能接受的是：

1. `planningPhase = COMPLETED`
2. 但文档没有任何变化

因此 v1 在交互上必须保证：

1. 待审批时，计划内容必须完整可读
2. 计划中必须显示：
   - 文档
   - 目标位置
   - 资产来源
   - 操作类型
3. 审批通过后，如果只是“计划通过”但尚未执行，不能返回 `COMPLETED`
4. 只有真实落盘成功，才能返回 `COMPLETED`
5. 如果缺图片来源、缺生成描述、缺定位结果，必须明确提示用户补充，而不是伪完成

### 5.6 用户端应看到的状态机

建议富媒体操作的用户态至少区分为：

1. `PLANNING`
   - 正在解析文档、锚点、资产
2. `WAITING_APPROVAL`
   - 计划已生成，等待用户确认
3. `EXECUTING`
   - 正在执行上传、创建节点、写入文档
4. `COMPLETED`
   - 真实写入成功
5. `FAILED`
   - 执行失败
6. `NEED_CLARIFICATION`
   - 资产来源或锚点不明确，需要补充

当前最大问题是：

1. `WAITING_APPROVAL -> COMPLETED`

之间少了真实 `EXECUTING` 语义。

## 6. v1 的总体整改方案

v1 采用“**文本 patch 链路保留，富媒体 execution-plan 链路补齐**”的双通道方案。

### 5.1 保留文本链路

文本类编辑继续走：

1. `DocumentPatchCompiler`
2. `DocumentPatchExecutor`
3. `DocumentTargetStateVerifier`

### 5.2 补齐富媒体链路

富媒体/布局类编辑改走：

1. `DocumentEditIntentResolver`
2. `AssetResolutionFacade`
3. `DocumentStructureSnapshotBuilder`
4. `DocumentAnchorResolver`
5. `DocumentEditStrategyPlanner`
6. `RichContentExecutionPlanner`
7. `RichContentExecutionEngine`
8. `RichContentTargetStateVerifier`

这不是与当前架构冲突，而是把当前已经搭好的富媒体支路真正接上电。

## 6. 必须新增的核心组件

### 7.1 `RichContentExecutionEngine`

这是 v1 最核心的新增组件。

建议位置：

1. `harness/document/iteration/support/RichContentExecutionEngine.java`

职责：

1. 读取 `DocumentEditPlan.executionPlan`
2. 顺序执行每个 `ExecutionStep`
3. 收集中间执行产物
4. 输出富媒体执行结果
5. 在失败时中止并返回明确错误

要求：

1. 不能复用 `DocumentPatchExecutor`
2. 不能把 step 执行写回 `DefaultDocumentIterationExecutionService`
3. 必须是独立执行引擎

建议输出：

#### `RichContentExecutionResult`

字段建议：

1. `createdBlockIds`
2. `createdAssetRefs`
3. `beforeRevision`
4. `afterRevision`
5. `runtimeArtifacts`
6. `stepResults`

### 7.2 `ExecutionStepHandlerRegistry`

建议位置：

1. `harness/document/iteration/support/ExecutionStepHandlerRegistry.java`

职责：

1. 将 `stepType` 或 `toolBinding` 路由到具体 handler

建议 handler 至少包括：

1. `UploadImageStepHandler`
2. `InsertImageBlockStepHandler`
3. `InsertTableBlockStepHandler`
4. `WriteTableDataStepHandler`
5. `CreateWhiteboardStepHandler`
6. `InsertWhiteboardRefStepHandler`
7. `SnapshotVerifyStepHandler`

这样可以避免 `RichContentExecutionEngine` 内部堆巨大 `switch`。

### 7.3 `RichContentExecutionContext`

职责：

1. 保存多步执行过程中的中间产物
2. 例如：
   - 上传后的 file token
   - 新建 whiteboard token
   - 新增 block id
   - table node block id

否则后续 step 无法消费前序结果。

## 8. 必须修改的现有组件

### 8.1 `DefaultDocumentIterationExecutionService`

这是 v1 的第一关键修改点。

必须改成：

1. 文本类动作：
   - 继续走 `patchExecutor`
2. 富媒体类动作：
   - 改走 `RichContentExecutionEngine`

建议新增分流函数：

1. `isRichExecutionPlan(editPlan)`
2. `executePatchPlan(...)`
3. `executeRichContentPlan(...)`

要求：

1. 审批通过后不能再统一走 `patchExecutor.execute(...)`
2. `COMPLETED` 只能在真实执行成功后返回

### 8.2 `DocumentEditIntentResolver`

当前必须补齐 `assetSpec` 输出。

要求：

1. 富媒体请求进入时，必须产出结构化 `MediaAssetSpec`
2. 不允许业务硬编码
3. 不允许 `contains("图片")` / `contains("表格")` / `contains("白板")`

正确做法：

1. 用 LLM 结构化输出资产规格
2. 输出字段至少包括：
   - `assetType`
   - `sourceType`
   - `sourceRef`
   - `caption`
   - `altText`
   - `generationPrompt`
   - `tableSchema`
   - `whiteboardDsl`

v1 对图片建议策略：

1. 如果用户没给来源，先允许走“生成占位图/生成示意图”路径
2. 但这必须由结构化 `sourceType=AI_GENERATED` 显式表达
3. 不能在 Java 里偷写默认图片逻辑

### 8.3 `DocumentAnchorResolver`

当前要让富媒体插入至少稳定落到 section/block 锚点。

要求：

1. 对 `INSERT_IMAGE_AFTER_ANCHOR`
2. 对 `INSERT_TABLE_AFTER_ANCHOR`
3. 对 `INSERT_WHITEBOARD_AFTER_ANCHOR`

都必须能从 snapshot 中解析出稳定插入锚点。

v1 重点：

1. 优先把“在某章节后插入”解析到 `SectionAnchor`
2. 再明确映射到“插在该 section heading 后”还是“插在该 section 尾 block 后”
3. `toPlanVO()` 里必须能看到 `targetTitle / targetPreview`

禁止：

1. 无锚点时默认为空还继续返回可执行计划
2. 锚点没解出来却继续返回 `COMPLETED`

### 8.4 `AssetResolutionFacade`

v1 不需要一步做到完美，但必须真正可执行。

建议：

1. 图片：
   - 已有外链 / 现有文件 / AI 生成 三类解析闭环
2. 表格：
   - 结构化 schema 生成功能可继续走 LLM
   - 但必须输出稳定 `TableModel`
3. 白板：
   - 至少输出 `whiteboardDsl` 或 `assetRef`

要求：

1. 如果资产无法解析，应该失败或转审批
2. 不能返回半空 `ResolvedAsset` 再继续执行

### 8.5 `RichContentTargetStateVerifier`

当前 verifier 必须改。

问题：

1. 它目前从 `patchOperations` 推导新 block id
2. 富媒体链路本来就不靠 patch

必须改成：

1. 从 `RichContentExecutionResult` 或 `RichContentExecutionContext` 读取运行时创建的 block id
2. 再基于 after snapshot 校验

v1 校验标准：

1. 图片：
   - 新 block id 存在
   - block 位于目标锚点后
2. 表格：
   - 新 table block 存在
   - 行列已写入
3. 白板：
   - 引用 block 存在
   - 关联 whiteboard token 已创建

### 8.6 `DocumentStructureSnapshotBuilder`

v1 至少要补一个最小增强：

1. 把富媒体 block 的原始类型和必要元数据带到 snapshot

原因：

1. 否则 verifier 只能做极弱校验
2. 无法区分普通 block 与媒体 block

不要求 v1 一步做到完整富节点树，但至少要补：

1. `rawPlatformPayload`
2. `nodeType` 或等价字段
3. 对 `media/table/embed/whiteboard` 的基本识别

## 9. v1 的执行链路设计

v1 推荐固定走四段式执行链：

### 9.1 Asset Resolve

输入：

1. `MediaAssetSpec`

输出：

1. `ResolvedAsset`

### 9.2 Asset Materialize

图片：

1. 上传文件或生成图片
2. 得到 file token / media ref

表格：

1. 生成 `TableModel`
2. 预备表头和数据

白板：

1. 创建 whiteboard
2. 得到 whiteboard token

### 9.3 Document Node Insert

执行：

1. 在正确锚点后插入媒体节点
2. 返回新 block id

### 9.4 Target State Verify

验证：

1. block 是否真实落盘
2. 位置是否正确
3. 类型是否符合预期
4. revision 是否推进

## 10. v1 推荐支持的 step 集合

为了避免一开始做太多，v1 只支持以下 step：

### 10.1 图片

1. `UPLOAD_IMAGE`
2. `INSERT_IMAGE_BLOCK`

## 11. v1 明确不做的错误修法

以下做法全部禁止：

1. 用 `append markdown image` 模拟图片插入
2. 用 `str_replace` 塞一个图片链接
3. 用 `block_replace` 拼一段“图片说明文字”
4. 用普通 paragraph 模拟白板引用
5. 用全文 markdown 重写模拟表格

   - 图片
   - 表格
   - 白板
   - 某章节后
7. 锚点为空仍继续生成可执行 plan
8. 审批通过但没执行仍返回 `COMPLETED`

如果你是 Claude，请按以下口径执行：
