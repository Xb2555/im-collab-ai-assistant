# 0504_docx 支持插入富媒体

## 1. 文档定位

本文是给 Claude 直接用于改代码的实施型 prompt，不是讨论稿。

目标：

1. 在当前场景 C 文档迭代架构基础上，增加对富媒体与画布操作的正式支持。
2. 支持在文档中通过自然语言指令插入和处理图片、表格、白板等富媒体内容，并支持结构化布局调整。
3. 保持与当前项目 `planner / harness / skills / store` 架构一致。
4. 禁止把新能力做成一堆 `if/else` 和字符串 patch 特例。
5. 禁止项目代码中出现任何业务硬编码。

一句话定义：

**必须把 docx/document iteration 从“文本编辑系统”升级为“面向多模态文档节点的结构编辑系统”。**

---

## 2. 最高优先级约束

这部分要求优先级最高，Claude 在实施时必须严格遵守。

### 2.1 禁止业务硬编码

禁止出现以下实现：

1. `if (text.contains("开头"))`
2. `if (text.contains("小节"))`
3. `if (text.contains("章节"))`
4. `if (text.equals("图片"))`
5. `if (text.equals("表格"))`
6. `if ("document_head".equals(...))`
7. `if ("whiteboard".equals(...))`
8. 任何靠中文词面直接决定：
   - `semanticAction`
   - `anchorType`
   - `strategyType`
   - `expectedState`
   - `assetType`

允许保留的只有：

1. 平台协议枚举
2. 通用 XML / Markdown / block 结构解析
3. 通用节点类型枚举
4. 通用文本规范化工具
5. LLM prompt 中的示例文本

说明：

1. 业务语义必须通过结构化 LLM 输出或显式 schema 解析得到。
2. fallback 只能返回 unresolved / clarification / approval required / unsupported。
3. 禁止“猜不到就用默认位置、默认章节、默认文首、默认正文”。

### 2.2 禁止堆积代码

不要做成：

1. `document` 模块里再加几十个 `if (assetType == ...)`
2. `patchCompiler` 里为图片、表格、白板各写一套散乱特判
3. 用全文 markdown 重写模拟布局操作
4. 用字符串拼接模拟媒体节点插入

必须做成：

1. 明确的数据模型
2. 明确的层次分工
3. 明确的策略矩阵
4. 明确的执行计划模型
5. 明确的目标状态校验模型

## 3. 为什么不能沿用当前纯文本迭代思路

图片、表格、画布、布局，不是“几种特殊文本”。

它们本质上是不同类型的文档节点：

1. `HeadingNode`
2. `ParagraphNode`
3. `ImageNode`
4. `TableNode`
5. `CalloutNode`
6. `DividerNode`
7. `EmbedNode`
8. `WhiteboardNode`
9. `LayoutContainerNode`

因此：

1. 富媒体插入不是 `str_replace`
2. 表格编辑不是 markdown 文本替换
3. 白板操作不是插一段说明文字
4. 布局调整不是全文重写

如果继续沿用“文本片段 + patch 命令拼装”的旧思路，结果一定会退化成：

1. 大量字符串硬编码
2. 大量媒体特判
3. 大量平台能力绕路
4. 大量 verify 补丁

## 4. 新能力的总架构

在当前 document iteration 六层基础上，升级为七层：

1. `Intent Layer`
2. `Asset Understanding Layer`
3. `Document Snapshot Layer`
4. `Anchor Resolution Layer`
5. `Edit Strategy Layer`
6. `Execution Planning Layer`
7. `Target State Verification Layer`

其中新增的关键是：

1. `Asset Understanding Layer`
2. `Execution Planning Layer`

原因：

1. 富媒体内容通常不是一次 patch 能完成，而是多步执行。
2. 文档编辑器不应该自己处理图片二进制、白板渲染、表格推导细节。

## 5. 核心设计原则

### 5.1 编辑语义优先于工具命令

先得到：

1. `semanticAction`
2. `assetSpec`
3. `targetAnchorSpec`
4. `layoutIntent`

不能先猜：

1. 要不要 `append`
2. 要不要 `block_replace`
3. 要不要 `str_replace`

### 5.2 统一节点模型

文本、图片、表格、白板、布局都统一纳入文档节点系统。

### 5.3 资产处理与文档编辑分层

文档编辑器负责：

1. 理解编辑目标
2. 定位锚点
3. 规划结构操作
4. 执行文档节点操作
5. 验证最终状态

资产子系统负责：

1. 图片生成 / 上传 / 引用
2. 表格 schema 推导 / 数据规整
3. 白板 DSL 生成 / 画布创建 / 更新

### 5.4 平台能力边界必须真实

如果平台不支持某种布局或某种媒体编辑：

1. 优先寻找合法结构等价变换
2. 其次进入审批或降级
3. 绝不允许用全文重写或字符串拼接伪造支持

### 5.5 成功判定按目标节点状态

verify 必须检查：

1. 节点是否存在
2. 节点位置是否正确
3. 节点引用是否正确
4. 节点结构是否保持稳定

不能只看：

1. 全文是否包含某行文本
2. revision 是否推进

## 6. 需要建立的核心数据模型

### 6.1 编辑语义模型

新增统一语义枚举，禁止针对富媒体单独散写分支。

建议至少支持：

1. `INSERT_IMAGE_AFTER_ANCHOR`
2. `REPLACE_IMAGE`
3. `DELETE_IMAGE`
4. `INSERT_TABLE_AFTER_ANCHOR`
5. `REWRITE_TABLE_DATA`
6. `APPEND_TABLE_ROW`
7. `DELETE_TABLE`
8. `INSERT_WHITEBOARD_AFTER_ANCHOR`
9. `UPDATE_WHITEBOARD_CONTENT`
10. `MOVE_BLOCK`
11. `MOVE_MEDIA_BLOCK`
12. `RELAYOUT_SECTION`
13. `CONVERT_TEXT_TO_TABLE`
14. `CONVERT_TEXT_TO_IMAGE_CARD`
15. `EXPLAIN_ONLY`

要求：

1. 语义稳定
2. 显式
3. 可审计
4. 不靠工具命令倒推

### 6.2 资产描述模型 `MediaAssetSpec`

不要把媒体信息塞进字符串 prompt 或 patch 里。

建议字段：

1. `assetType`
2. `sourceType`
3. `sourceRef`
4. `mimeType`
5. `title`
6. `altText`
7. `caption`
8. `dimensionsHint`
9. `renderIntent`
10. `generationPrompt`
11. `tableSchema`
12. `tableData`
13. `whiteboardDsl`
14. `layoutHints`

### 6.3 文档快照模型升级

现有 snapshot 不能只表达 heading/plainText，必须支持富节点：

建议 `DocumentStructureNode` 至少支持：

1. `nodeId`
2. `nodeType`
3. `blockId`
4. `parentId`
5. `children`
6. `textContent`
7. `mediaMeta`
8. `tableMeta`
9. `layoutMeta`
10. `positionIndex`
11. `topLevelSectionId`
12. `rawPlatformPayload`

### 6.4 富媒体锚点模型

锚点类型扩展为：

1. `TextAnchor`
2. `BlockAnchor`
3. `SectionAnchor`
4. `MediaAnchor`
5. `TableAnchor`
6. `LayoutAnchor`
7. `DocumentHeadAnchor`
8. `DocumentTailAnchor`

### 6.5 执行计划模型

不要直接生成 patch 列表，先生成执行计划图。

建议新增：

#### `ExecutionPlan`

字段建议：

1. `steps`
2. `dependencies`
3. `requiresApproval`
4. `rollbackPolicy`
5. `expectedState`

#### `ExecutionStep`

字段建议：

1. `stepType`
2. `input`
3. `output`
4. `toolBinding`
5. `retryPolicy`
6. `failureMode`

## 7. 七层架构的职责定义

### 7.1 Intent Layer

职责：

1. 把自然语言请求解析成结构化编辑意图
2. 输出：
   - `semanticAction`
   - `targetAnchorSpec`
   - `assetSpec`
   - `layoutIntent`
   - `constraints`
3. 不负责 patch 命令选型
4. 不负责 block id 定位
5. 不负责资产上传或创建

要求：

1. 必须走结构化 LLM 输出
2. 不允许本地代码用词面硬编码做二次意图推断

### 7.2 Asset Understanding Layer

职责：

1. 理解资产来自哪里
2. 将“图片/表格/白板请求”解析为可执行资产
3. 产出统一 `ResolvedAsset`

建议拆为：

1. `ImageAssetResolver`
2. `TableAssetResolver`
3. `WhiteboardAssetResolver`

说明：

1. 如果用户要插入图片，先解析图片来源，不要直接写文档。
2. 如果用户要把列表转成表格，先产出表格结构模型。
3. 如果用户要插入流程图白板，先产出白板 DSL 或画布定义。

### 7.3 Document Snapshot Layer

职责：

1. 构建统一结构快照
2. 包含文本节点与富媒体节点
3. 提供位置、父子关系、引用关系、布局容器关系

要求：

1. 后续锚点解析、策略规划、verify 全部基于同一份 snapshot
2. 不允许中途再随手 fetch 零散结果拼逻辑

### 7.4 Anchor Resolution Layer

职责：

1. 在快照上解析真实锚点
2. 按锚点类型区分文本、块、章节、媒体、表格、布局
3. 无法唯一定位时返回 unresolved

禁止：

1. `contains(snippet)` 直接猜 block
2. 关键词没命中就退到第一个章节
3. 业务词面硬编码推断“这是图片前/表格后”

### 7.5 Edit Strategy Layer

职责：

1. 把 `semanticAction + anchorType + assetType` 映射为唯一允许策略
2. 通过显式策略矩阵控制所有媒体能力

例如：

1. `INSERT_IMAGE_AFTER_ANCHOR`
   - 允许锚点：`BlockAnchor | SectionAnchor`
   - 策略：`create image block -> insert after`

2. `CONVERT_TEXT_TO_TABLE`
   - 允许锚点：`BlockAnchor | RangeAnchor`
   - 策略：`extract text -> resolve table asset -> insert table -> optional delete source`

3. `INSERT_WHITEBOARD_AFTER_ANCHOR`
   - 允许锚点：`BlockAnchor | SectionAnchor`
   - 策略：`create whiteboard -> create embed/ref block -> insert after`

4. `RELAYOUT_SECTION`
   - 允许锚点：`SectionAnchor | LayoutAnchor`
   - 策略：`reorder child nodes / move blocks / regroup containers`

要求：

1. 平台不支持时显式失败或审批
2. 绝不允许全文重写模拟布局

### 7.6 Execution Planning Layer

职责：

1. 将策略编译成多步执行计划
2. 负责跨工具协调
3. 负责依赖顺序和回滚策略

示例：

插入图片可能是：

1. 生成图片资产
2. 上传图片
3. 创建图片块
4. 插入文档
5. 更新 caption
6. 校验顺序与引用

列表转表格可能是：

1. 提取原始文本块
2. 推导 table schema
3. 生成 table asset
4. 插入表格
5. 删除原列表
6. 校验表头和行数

### 7.7 Target State Verification Layer

职责：

1. 按节点状态校验编辑成功
2. 不按全文字符串校验

图片校验：

1. 目标位置存在 `ImageNode`
2. 引用正确
3. caption/altText 正确
4. 顺序正确

表格校验：

1. 存在 `TableNode`
2. 行列数正确
3. 表头匹配
4. 数据已写入

白板校验：

1. 存在 `WhiteboardNode` 或等价 embed node
2. whiteboard token 正确
3. 引用成功

布局校验：

1. 节点顺序正确
2. 容器关系正确
3. 原节点未丢失

## 8. 三类富媒体能力的推荐实现方式

### 8.1 图片

图片支持三类来源：

1. 已有附件
2. 外部链接
3. AI 生成

正确实现：

1. `Intent Layer` 识别为插图语义
2. `Asset Layer` 解析来源并得到 `ResolvedImageAsset`
3. `Anchor Layer` 解析插入位置
4. `Strategy Layer` 规划图片节点插入
5. `ExecutionPlan` 协调上传 / 插入 / 更新 caption
6. `Verifier` 校验图片节点和位置

禁止：

1. 直接往 markdown 里拼图片占位文本
2. 用一段说明文字替代真实图片节点

### 8.2 表格

表格支持三类语义：

1. 自然语言生成表格
2. 列表/段落转表格
3. 修改已有表格数据

建议新增统一 `TableModel`：

1. `columns`
2. `rows`
3. `cellValues`
4. `formattingHints`

正确实现：

1. 先产出表格结构模型
2. 再插入表格节点
3. 再执行表格数据落盘

禁止：

1. 把表格当普通 markdown 文本块长期处理
2. 用全文改写模拟“转表格”

### 8.3 白板 / 画布

正确做法不是插一张静态图，而是：

1. 创建独立 whiteboard 实体
2. 文档中插入白板引用节点或 embed 节点
3. 白板内容由独立 whiteboard skill / service 管理

推荐流程：

1. 解析白板语义
2. 生成 whiteboard DSL
3. 调白板服务创建画布
4. 文档中插入引用
5. 校验引用与预览成功

禁止：

1. 用文本描述假装白板已插入
2. 用全文重写模拟图示结构

## 9. 布局调整的正确边界

布局调整最容易被做烂。

必须坚持：

1. 布局调整本质是结构节点重排或容器调整
2. 绝不能退化成全文 markdown 重写
3. 要显式建模 `LayoutIntent`

建议支持的语义：

1. `MOVE_BLOCK`
2. `MOVE_MEDIA_BLOCK`
3. `GROUP_AS_CARD`
4. `SPLIT_SECTION`
5. `MERGE_BLOCKS`
6. `REORDER_LIST`
7. `PROMOTE_TO_CALLOUT`
8. `RELAYOUT_SECTION`

如果平台不支持复杂双栏、自由布局等：

1. 明确降级成平台支持的卡片式结构
2. 或进入审批
3. 或建议转移到白板/PPT

禁止：

1. 全文重写“伪造布局”
2. 把布局调整偷偷变成文案改写

## 11. 推荐的最小可交付版本

第一阶段不要一次做太大，优先完成四个能力：

1. 在指定锚点后插入图片
2. 将列表/段落转换成表格
3. 在文档中插入白板引用
4. 调整 section 内 block 顺序

原因：

1. 覆盖富媒体插入
2. 覆盖结构化数据生成
3. 覆盖外部画布联动
4. 覆盖布局调整

同时不会把系统做成巨型黑箱。

一句话总结：

**必须把场景 C 从“文本 patch 系统”升级为“多模态文档节点编辑系统”，以结构快照、资产解析、执行计划和目标状态校验为中心实现富媒体与画布操作。**

## 13. 验收标准

本次架构设计与后续实现，至少要满足：

1. 本地 Java 代码不再依赖中文关键词硬编码决定富媒体业务语义。
2. 图片、表格、白板、布局都被纳入统一节点模型。
3. 文档编辑器不直接处理资产生成细节，而是依赖资产解析层。
4. 富媒体操作通过执行计划图驱动，不是单 patch 乱拼。
5. verify 以节点状态和结构状态为主，不是全文字符串判断。
6. 平台不支持的能力显式失败、审批或降级。
7. 第一阶段最小交付四项能力可以独立演示。

最终判断标准不是“代码加了多少类”，而是：

**架构是否真正稳定、可扩展、无硬编码、无伪实现。**
