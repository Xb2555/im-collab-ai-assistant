• 基于当前本机 lark-cli 1.0.23，富媒体与画布能力是可以做一版明确边界的实现，但不能泛化成“任意图片/表格样式编辑器”。

  我先给出已确认的 CLI 能力边界：

  - lark-cli docs +media-insert
    支持把本地图片或文件插入文档；默认追加到末尾，也支持基于文本选择插入。
  - lark-cli docs +media-upload
    支持把媒体文件上传到文档 block。
  - lark-cli docs +media-download / +media-preview
    支持读取文档里的媒体。
  - lark-cli docs +whiteboard-update
    支持更新文档中的已有 whiteboard，输入是 mermaid / plantuml / whiteboard dsl。
  - lark-cli docs +update
    仍然是文档文本/结构更新主命令。
  - 没看到文档侧独立的“创建表格块并逐单元格写值”的一等命令暴露在 help 里。
  - 也没看到文档侧“图片尺寸、对齐、环绕、边框、裁剪”这类样式 API 能力暴露在 help 里。

  所以实现方案必须按真实 CLI 能力来收口，而不是按产品想象来写。

  我建议把需求拆成 4 类能力：

  1. 图片插入
  2. 白板插入/更新
  3. 表格插入
  4. 布局调整

  其中结论很直接：

  - 图片插入：可以做
  - 白板更新：可以做
  - 白板插入：大概率可做，但要看当前 CLI/skill 是否已有“创建白板并嵌入文档”的现成封装
  - 表格插入：可以做基础版，但不要承诺“改单元格样式”
  - 布局调整：只能做结构位置调整，不能叫样式布局系统

  ———

  一、推荐的能力范围

  1. 图片
    支持：

  - 在章节/块后插入图片
  - 替换已有图片
  - 移动图片块到别的位置

  不支持或暂不承诺：

  - 调整尺寸
  - 对齐/环绕/边框/阴影
  - 裁剪/滤镜

  原因：
  CLI help 没暴露这些样式能力，硬做会退化成猜 patch。

  2. 表格
    支持：

  - 在章节/块后插入表格
  - 从结构化数据生成表格
  - 追加行
  - 重写整张表的数据

  不支持或暂不承诺：

  - 单元格级样式
  - 边框色、底纹、列宽、字体样式精细控制

  原因：
  当前可见能力更适合“整表生成/替换”，不适合细粒度样式编辑。

  3. 白板
    支持：

  - 更新已有 whiteboard 内容
  - 用 mermaid/plantuml/dsl 驱动更新

  条件支持：

  - 在文档中插入新的 whiteboard 引用
    前提是你们已有创建 whiteboard token 并插入引用块的链路，否则只能先做“更新已有白板”。

  4. 布局调整
    支持：

  - 图片/表格/白板块移动到某章节后
  - 同章节内媒体块重排
  - 章节级重排

  不支持：

  - 页面级版式
  - 浮动布局
  - 图文混排样式控制

  ———

  二、按 CLI 1.0.23 设计的实现方案

  ### 1. 图片插入链路

  实现方式：

  - 输入：
      - 本地文件路径 / 附件 / 已下载临时文件
      - 锚点：SectionAnchor 或 BlockAnchor
  - 执行：
      - 优先直接走 docs +media-insert
      - 如果必须精确定位到锚点后，而 +media-insert 不够稳定，则保留现有 upload + insert image block 双步链路作为内部执行
        器
  - verify：
      - after snapshot 中目标锚点后出现新 image/media block
      - 新 block 与目标锚点顺序正确
      - revision 推进

  建议语义：

  - INSERT_IMAGE_AFTER_ANCHOR
  - REPLACE_IMAGE
  - MOVE_MEDIA_BLOCK

  ### 2. 白板更新链路

  实现方式：

  - 输入：
      - 目标 whiteboard anchor
      - mermaid / plantuml / whiteboard dsl
  - 执行：
      - 直接走 docs +whiteboard-update
  - verify：
      - whiteboard block 仍存在
      - token 不变或合法更新
      - revision 推进
      - 可选：拉预览图或 DSL 回读校验

  建议语义：

  - UPDATE_WHITEBOARD_CONTENT

  ### 3. 白板插入链路

  如果当前 skill/CLI 已支持：

  - create whiteboard
  - 拿到 whiteboard token
  - 插入文档 whiteboard ref block

  那就实现：

  - INSERT_WHITEBOARD_AFTER_ANCHOR

  否则先不要对外承诺“插入白板”，只承诺“更新已有白板”。

  ### 4. 表格插入与更新链路

  我建议分两档：

  基础版

  - 把结构化数据渲染成 markdown table
  - 通过文档插入能力插入到锚点后
  - 或用当前已有 InsertTableBlockStepHandler / WriteTableDataStepHandler 走内部链路

  稳定版

  - 明确引入 TableModel
      - columns
      - rows
      - caption
      - sourceType
  - 执行时始终以“整表生成/整表替换”为主，不做单元格补丁

  verify：

  - after snapshot 中出现 table block
  - 表头文本存在
  - 行数或关键单元格内容符合预期

  建议语义：

  - INSERT_TABLE_AFTER_ANCHOR
  - REWRITE_TABLE_DATA
  - APPEND_TABLE_ROW

  三、代码层应该怎么落

  你们当前代码已经有一半骨架了，应该继续沿这个方向，而不是重起炉灶。

  ### 1. Intent Layer

  继续保留这些语义枚举：

  - INSERT_IMAGE_AFTER_ANCHOR
  - REPLACE_IMAGE
  - INSERT_TABLE_AFTER_ANCHOR
  - REWRITE_TABLE_DATA
  - APPEND_TABLE_ROW
  - INSERT_WHITEBOARD_AFTER_ANCHOR
  - UPDATE_WHITEBOARD_CONTENT
  - MOVE_MEDIA_BLOCK
  - RELAYOUT_SECTION

  但 prompt 要明确：

  - 图片/表格/白板是三类不同资产
  - “调整布局”不是“调整样式”
  - 没有明确资产来源时要澄清

  ### 2. Asset Layer

  这是现在最需要补强的层。

  建议统一成：

  - ResolvedImageAsset
  - ResolvedTableAsset
  - ResolvedWhiteboardAsset

  至少要支持：

  图片：

  - ATTACHMENT
  - LOCAL_FILE
  - EXTERNAL_URL

  表格：

  - INLINE_TABLE_DATA
  - CSV
  - MARKDOWN_TABLE
  - STRUCTURED_ROWS

  白板：

  - WHITEBOARD_DSL
  - MERMAID
  - PLANTUML
  - WHITEBOARD_TOKEN

  ### 3. Anchor Layer

  富媒体也必须走你现在刚收好的确定性锚点体系：

  - 插入图片/表格：SectionAnchor 或 BlockAnchor
  - 更新图片：MediaAnchor
  - 更新表格：TableAnchor
  - 更新白板：WhiteboardAnchor

  重点：
  不要再靠 blockType.contains("table") 这种弱识别做主路径。
  如果 snapshot 元数据不够，就扩 snapshot。

  ### 4. Execution Layer

  建议明确分流：

  - 文本/章节编辑：
    DocumentPatchCompiler + DocumentPatchExecutor
  - 富媒体/白板：
    RichContentExecutionPlanner + RichContentExecutionEngine

  不要把图片/表格继续塞回 block_replace 主路径。

  ### 5. Verify Layer

  每类资产单独验：

  - EXPECT_IMAGE_NODE_PRESENT
  - EXPECT_TABLE_NODE_PRESENT
  - EXPECT_WHITEBOARD_NODE_PRESENT
  - EXPECT_MEDIA_NODE_MOVED

  这部分你们 docs 里其实已经有方向了，直接落地即可。

  ———

  四、我对“是否已完成”的判断

  按 CLI 1.0.23 的真实能力，我的判断是：

  - 图片插入：可以做成稳定能力
  - 白板更新：可以做成稳定能力
  - 表格插入/整表更新：可以做成基础稳定能力
  - 细粒度样式调整：现在不该承诺
  - 复杂布局操作：只能做块级/章节级重排，不是完整版式系统

  ———

  五、建议的实施优先级

  按最容易闭环的顺序做：

  1. 图片插入
     原因：CLI 已明确有 docs +media-insert
  2. 白板更新
     原因：CLI 已明确有 docs +whiteboard-update
  3. 表格插入与整表更新
     原因：虽然样式不行，但结构化插入可做
  4. 白板插入
     前提：确认是否已有 whiteboard create + embed 链路
  5. 媒体块布局重排
     原因：依赖稳定锚点和块 identity
