# 1. PlannerController 任务入口

用户从 GUI 发起一次“帮我生成一篇 spring ai alibaba 技术文档”时，请求首先进入 `POST /planner/tasks/plan`。这一层只负责接收请求、绑定当前登录用户到 `WorkspaceContext`，然后把任务提交给异步 planner。

相关类：

- [PlannerController](C:/Users/14527/Desktop/lark-ai/project/im-collab-ai-assistant/im-collab-ai-assistant-server/im-collab-ai-assistant-app/src/main/java/com/lark/imcollab/app/planner/controller/PlannerController.java)

关键动作：

- 读取 `PlanRequest.rawInstruction`
- 注入 `WorkspaceContext.senderOpenId`
- 注入 `WorkspaceContext.inputSource=GUI`
- 调用 `AsyncPlannerService.submitPlan(...)`

# 26. 当前“生成 spring ai alibaba 技术文档”链路的架构总结

当前代码里，这个任务不是“planner 直接调大模型写文档”，而是下面这条分层链路：

1. `PlannerController` 接单
2. `AsyncPlannerService` 建立异步规划任务
3. `TaskSessionResolver + TaskIntakeService` 判定这是一个新文档任务
4. `PlannerConversationService` 封装会话
5. `PlannerSupervisorGraphRunner / PlannerSupervisorGraphNodes` 完成场景 B 规划编排
6. `TaskPlanningService + ExecutionContractFactory + TaskBridgeService` 生成可执行 `WRITE_DOC` 任务
7. `DefaultImTaskCommandFacade + DefaultHarnessFacade` 确认执行并切到 harness
8. `ExecutionOrchestrator + ExecutionTaskDispatcher` 分发到文档执行器
9. `DefaultDocumentExecutionService + documentWorkflow` 执行场景 C 文档生成图
10. `DocumentWorkflowNodes` 依次完成大纲、分节、审稿、装配、飞书落盘
11. `DocumentExecutionSupport + TaskRuntimeProjectionService` 回写 artifact 和 runtime 视图

这说明本周期场景 B 和场景 C 的架构设计，当前代码已经形成了比较清晰的两段式结构：

- 场景 B：`planner supervisor graph` 负责理解、规划、审查、gate、runtime 投影
- 场景 C：`document workflow graph` 负责文档大纲、章节生成、审稿、渲染、落盘

但从当前实现看，也有几个明显的现实状态：

- planner 到 harness 的桥接已经成型，边界在 `TaskBridgeService`
- 文档生成已经不是整篇黑盒生成，而是 `DocumentPlan -> SectionDraft -> Review -> Render -> CreateDoc`
- 飞书文档落盘入口已经统一收敛到 `LarkDocTool.createDoc(...)`
- 图表能力的节点已经预留，但 `ExecutionContractFactory.resolveDiagramRequirement()` 当前固定返回 `required=false`，所以 Mermaid 目前默认不会真正强制启用
