# 任务中断与调整问题分析

## 现象描述

用户反馈：
1. 任务执行中断后，任务直接消失了（而不是停下来调整）
2. 停下来调整后，会把旧任务的产物返回（而不是基于旧产物继续修改）

---

## 结论

经过代码分析，发现：

### 问题一：任务中断后"消失了"

**根因：设计如此，而非 BUG**

当前代码设计为「中断后重规划」：

```java
// PlannerConversationService.java:573
sessionService.markAborted(session.getTaskId(), 
    "User interrupted execution for IM plan adjustment: " + graphInstruction);
```

当用户发出调整指令时，代码流程是：

1. **cancelExecution** - 中断正在执行的任务
2. **markAborted** - 将任务标记为 `ABORTED` 状态（终止态）
3. **replan** - 创建全新的任务（new taskId 或复用原 taskId 重新跑）

这意味着：
- 旧任务的执行被**完全终止**，不是暂停
- 旧任务的产物**不会自动继承**到新任务
- 用户需要在重规划时**手动指定**目标产物

**预期用户行为 vs 实际用户行为**

| 预期 | 实际 |
|------|------|
| 任务暂停，等待调整 | 旧任务被标记 ABORTED |
| 基于旧产物继续修改 | 需要明确指定目标产物 ID |

### 问题二：调整时返回旧产物

**根因：目标产物 ID 解析问题**

当用户说"把第三页改一下"时，代码需要解析出：
1. 要修改哪个产物（artifact ID）
2. 产物是否仍然存在于 Runtime 中

代码逻辑（`ReplanNodeService.java:116-127`）：

```java
if (hasText(extractTargetArtifactId(adjustmentInstruction))) {
    session.setPlanningPhase(PlanningPhaseEnum.COMPLETED);
    return replanCompletedTask(session, adjustmentInstruction, workspaceContext);
}
if (isCompletedArtifactContext(taskId, session)) {
    session.setPlanningPhase(PlanningPhaseEnum.COMPLETED);
    return replanCompletedTask(session, adjustmentInstruction, workspaceContext);
}
if (session.getPlanningPhase() == PlanningPhaseEnum.COMPLETED) {
    return replanCompletedTask(session, adjustmentInstruction, workspaceContext);
}
```

**问题在于**：如果产物解析失败或产物不存在，代码可能：
1. 去 Runtime 查已有产物列表
2. 找不到匹配目标时，返回**旧的产物**（用户看到的"返回旧产物"）

---

## 关键代码路径

### 中断调整流程

```
用户发送调整指令 
    ↓
shouldRouteAdjustment() → true
    ↓
executionTool.cancelExecution(taskId, "interrupt execution for plan adjustment")
    ↓
sessionService.markAborted(taskId, "User interrupted execution...")
    ↓
graphRunner.run(PLAN_ADJUSTMENT, taskId, instruction)
    ↓
ReplanNodeService.replan()
    ↓
如果没指定目标产物 → 进入 editExistingArtifact 或重新生成
```

### 产物返回逻辑

```
调整指令 → extractTargetArtifactId()
    ↓ (如果没找到明确 ID)
findEditableArtifacts()
    ↓ (如果产物存在但 ID 不匹配)
可能返回旧产物或要求用户选择
```

---

## 设计问题总结

| 问题 | 根因 | 影响 |
|------|------|------|
| 中断后任务消失 | 设计为`markAborted`而非暂停 | 用户感知任务终止 |
| 调整返回旧产物 | 目标产物 ID 解析失败 | 产物不匹配 |
| 需要手动指定产物 | 未从上下文中自动推断 | 用户体验差 |

---

## 改进建议

### 建议一：区分"暂停"和"取消"

当前只支持「取消 + 重规划」，建议增加「暂停」模式：

```java
// 暂停模式
session.setPlanningPhase(PlanningPhaseEnum.PAUSED);  // 新增状态
// 而不是
session.setPlanningPhase(PlanningPhaseEnum.ABORTED);  // 当前
```

### 建议二：自动继承旧产物

在 `replan` 时自动带产物上下文：

```java
// 自动从 Runtime 继承产物
PlanTaskSession resume = sessionService.get(taskId);
ArtifactRecord oldArtifact = findLatestArtifact(taskId);
if (oldArtifact != null) {
    instruction = appendTargetArtifact(instruction, oldArtifact.getArtifactId());
}
```

### 建议三：更智能的产物匹配

从用户意图中推断目标产物：

- 用户说"第三页" → 解析为 PPT 的第 3 页
- 用户说"刚才那个文档" → 最近生成的 DOC 产物
- 用户说"继续改" → 上一个产物

---

## 结论

1. **任务"消失"是设计行为**，当前为 `ABORTED` + 重规划，不是暂停
2. **返回旧产物是 BUG**，目标产物 ID 解析或查找失败
3. 两者都会导致用户感知上的"不顺"，建议增加「暂停」模式和「自动继承」逻辑
