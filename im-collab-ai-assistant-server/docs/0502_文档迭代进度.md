  本轮对话后，最重要的现状
  现在最大的剩余问题，不是架构没搭好，而是：

  真实飞书文档上，block_replace 的返回语义和最初假设不一致，导致“写成功但后端误判失败”。

  真实现象已经被验证过多次：

  1. 用户请求：
      - 在文章开头加一个章节；一、项目背景
  2. 飞书文档内容实际已经被写进去
  3. 但后端抛异常：
      - block_replace 后既未保留原 block，也未检测到替换结果，校验失败
  4. 这说明飞书在某些 block_replace 场景下会出现：
      - revision 已推进
      - 内容已落盘
      - 原 block id 消失
      - new_blocks 为空或不稳定
  5. 之前的校验逻辑把这种情况误判成失败

  这个问题在源码里已经继续修过
  最新修正思路是：

  - block_replace 成功判定以“内容证据 + revision 推进”为主
  - original block id 和 new_blocks 只作为辅助证据
  - 不再因为“没保留旧 block / 没返回 new_blocks”直接失败

  也就是说，下面这种错误分支应该已经从源码逻辑上被降级：

  - block_replace 后既未保留原 block，也未检测到替换结果

  但是用户仍然反馈运行中的服务报同样异常
  这说明有两种高概率情况：

  1. 当前运行中的服务还是旧进程，没加载最新 class
  2. 或者还有另一条未覆盖的 verifyBlockReplace 失败路径

  从对话看，第一种概率更高，因为本地源码的 harness 单测和 app compile 都通过了。
