# 下一步推荐前端接入契约

## 1. 目标

前端需要支持两件事：

1. 展示已完成任务的“下一步推荐”
2. 点击某条推荐后，触发后端继续规划或执行

这套能力只依赖两个接口：

- `GET /api/planner/tasks/{taskId}/runtime`
- `POST /api/planner/tasks/{taskId}/recommendations/{recommendationId}/execute`

## 2. 鉴权

如果服务开启鉴权，请带：

```http
Authorization: Bearer <token>
```

如果测试环境关闭了 `PLANNER_AUTH_ENABLED`，可不带。

## 3. 读取推荐

### 3.1 接口

```http
GET /api/planner/tasks/{taskId}/runtime
```

### 3.2 成功响应关键字段

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "task": {
      "taskId": "xxx",
      "version": 6,
      "status": "COMPLETED"
    },
    "evaluation": {
      "verdict": "PASS",
      "suggestions": [],
      "nextStepRecommendations": [
        {
          "code": "GENERATE_SHAREABLE_SUMMARY",
          "recommendationId": "GENERATE_SHAREABLE_SUMMARY",
          "title": "生成一段可直接发送的任务摘要",
          "reason": "当前主要产物已经齐备，下一步适合整理成可直接转发到群里或私聊里的摘要文本。",
          "actionLabel": "生成摘要",
          "executable": true,
          "suggestedUserInstruction": "基于当前任务内容生成一段可直接发送的摘要",
          "targetDeliverable": "SUMMARY",
          "followUpMode": "CONTINUE_CURRENT_TASK",
          "targetTaskId": "xxx",
          "sourceArtifactId": "artifact-xxx",
          "sourceArtifactType": "DOC",
          "plannerInstruction": "保留现有产物，新增一段可直接发送的任务摘要。",
          "artifactPolicy": "KEEP_EXISTING_CREATE_NEW",
          "priority": 1
        }
      ]
    }
  }
}
```

### 3.3 前端展示条件

只在以下条件同时满足时展示推荐区：

- `data.task.status === "COMPLETED"`
- `data.evaluation != null`
- `data.evaluation.verdict === "PASS"`
- `data.evaluation.nextStepRecommendations.length > 0`

### 3.4 推荐项字段说明

前端重点使用这些字段：

- `recommendationId`
  - 推荐动作唯一标识
  - 点击按钮时必须用它发起执行
- `title`
  - 卡片标题
- `reason`
  - 卡片说明
- `actionLabel`
  - 主按钮文案，直接展示
- `executable`
  - 是否允许点击
- `targetDeliverable`
  - 可做 tag，如 `DOC` / `PPT` / `SUMMARY`
- `priority`
  - 数字越小越靠前展示

### 3.5 不建议前端依赖的字段

这些字段可以展示参考，但不要当协议主键：

- `suggestedUserInstruction`
- `plannerInstruction`
- `artifactPolicy`

前端不要自己拼自然语言去调用旧的 replan 接口。

## 4. 执行推荐

### 4.1 接口

```http
POST /api/planner/tasks/{taskId}/recommendations/{recommendationId}/execute
Content-Type: application/json
```

### 4.2 请求体

最小请求体：

```json
{
  "version": 6
}
```

可选带上下文：

```json
{
  "version": 6,
  "workspaceContext": {
    "selectedMessages": ["补充上下文A", "补充上下文B"]
  }
}
```

### 4.3 请求字段说明

- `version`
  - 必填
  - 使用当前任务的版本号
  - 建议取自：
    - `/planner/tasks/{taskId}` 返回的 `data.version`
    - 或 `/runtime` 里的 `data.task.version`
- `workspaceContext`
  - 可选
  - 未来如果前端支持“选消息/选文档后继续”，可透传

### 4.4 成功响应

返回 `PlanPreviewVO`：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "taskId": "xxx",
    "version": 7,
    "planningPhase": "PLAN_READY",
    "title": "生成前10分钟消息总结文档、基于总结文档生成5页汇报PPT（每页约50字）和任务摘要（可直接发送）",
    "summary": "...",
    "cards": [...],
    "actions": {
      "canConfirm": true,
      "canReplan": true,
      "canCancel": true
    },
    "assistantReply": null
  }
}
```

### 4.5 成功后的前端处理

收到 `code === 0` 后，前端应当：

1. 立刻用返回的 `PlanPreviewVO` 更新当前工作台视图
2. 立即隐藏本地推荐区
3. 如果返回的 `data.taskId` 与当前不同，切换到新任务
4. 随后继续：
   - 订阅该任务 SSE
   - 或重新拉 `/planner/tasks/{taskId}`
   - 或重新拉 `/runtime`

### 4.6 重要约定

执行成功后，前端不要继续显示旧推荐按钮。

即使本地还缓存着旧的 `nextStepRecommendations`，也应在点击成功后立刻清空。

## 5. 状态约定

### 5.1 推荐区状态判断

前端推荐区的唯一判断源建议是 `/runtime`：

- `task.status === "COMPLETED"` 且 `evaluation != null`：可展示推荐
- 只要 `task.status !== "COMPLETED"`：不要展示推荐

### 5.2 执行后的状态切换

执行推荐后，任务通常会离开 completed，进入下一轮状态，例如：

- `PLAN_READY`
- `EXECUTING`
- 后续再次 `COMPLETED`

前端可这样理解：

- `/recommendations/.../execute` 的返回体，是“点击后立即切换视图”的主依据
- `/runtime` 是“详情页长期刷新”的主依据

### 5.3 一个实际注意点

执行成功后，`/planner/tasks/{taskId}` 和 `/runtime` 的状态语义不完全同名：

- `PlanPreviewVO.planningPhase` 可能是 `PLAN_READY`
- `runtime.task.status` 可能对应 `WAITING_APPROVAL`

前端处理建议：

- 工作台主流程状态：看 `PlanPreviewVO.planningPhase`
- 推荐区是否显示：只看 `runtime.task.status === COMPLETED`

## 6. 错误码约定

### 6.1 成功

```json
{ "code": 0, "message": "ok" }
```

### 6.2 版本冲突

```json
{ "code": 40900, "message": "版本冲突，请刷新后重试" }
```

前端处理：

- toast 提示“任务已更新，请刷新后重试”
- 重新拉 `/planner/tasks/{taskId}` 和 `/runtime`

### 6.3 推荐已过期或不可执行

```json
{
  "code": 50001,
  "message": "Recommendation not found or not executable: GENERATE_SHAREABLE_SUMMARY"
}
```

前端处理：

- 提示“该推荐已失效，请刷新任务状态”
- 重新拉 `/runtime`
- 清空旧推荐区

### 6.4 任务不存在或无权限

```json
{ "code": 40400, "message": "Task not found: xxx" }
```

### 6.5 未登录

```json
{ "code": 40100, "message": "Not logged in" }
```

## 7. 前端推荐接入流程

### 7.1 展示流程

1. 打开任务详情页
2. 调 `/api/planner/tasks/{taskId}/runtime`
3. 满足 completed + pass + recommendations 非空时展示推荐卡片
4. 按 `priority` 升序展示

### 7.2 点击流程

1. 用户点击某条推荐按钮
2. 禁用全部推荐按钮，进入 loading
3. 调 `POST /api/planner/tasks/{taskId}/recommendations/{recommendationId}/execute`
4. 成功：
   - 使用返回的 `PlanPreviewVO` 更新页面
   - 立即隐藏推荐区
   - 继续刷新任务详情
5. 失败：
   - 如果 `40900`：刷新任务
   - 如果 `50001`：刷新 runtime，并移除旧推荐
   - 其他错误：toast

## 8. 前端最小实现建议

推荐卡片最少展示：

- 标题：`title`
- 说明：`reason`
- 标签：`targetDeliverable`
- 主按钮：`actionLabel`
- 禁用态：`!executable`

点击时只传：

- `taskId`
- `recommendationId`
- `version`

## 9. 一句话原则

前端把“下一步推荐”当成结构化动作来接，不要把它当成“替用户发一句自然语言”。
