## 3. 当前项目背景与原始需求

### 3.1 原始题目背景

当前产品不是一个“单轮生成文本”的 demo，而是《基于 IM 的办公协同智能助手》。

它要求：

1. 文档、PPT、白板、IM 协同能力可以自由组合编排。
2. 必须有可视化任务状态。
3. 必须支持动态重规划、局部回退、人机协作、审批门禁。
4. 文档生成不是孤立能力，而是整个 harness 执行系统中的一种能力型工作流。

### 3.2 `im_harness架构设计.md` 的核心思想

本次整改必须严格遵守以下原则，不得违反。

1. 整个系统唯一的大脑层是 `Planner / Orchestrator`，不能再偷偷长出第二个“总控脑”。
2. 编排对象应该是“步骤图 / 状态机”，不是固定顺序的黑箱多 Agent 流水线。
3. Gate 必须分阶段设置，不要靠最后一个统一评审节点兜底所有问题。
4. 能工具化的能力尽量工具化，不要把所有能力人格化成 Agent。
5. 失败优先局部回退，不整轮从头重跑。
6. 多端同步、任务状态、Artifact 沉淀是第一公民。

### 3.3 当前项目里文档生成能力的真实实现形态

当前文档生成主链路大致是：

1. `generateOutline`
2. `generateSections`
3. `reviewDoc`
4. `writeDocAndSync`

其中：

1. `generateSections` 是按 outline 的 sections 做串行生成，不是并行写飞书。
2. 各章节生成时，只拿到当前 heading、keyPoints 和 task context，看不到前后章节正文。
3. 最终写飞书前，不是“按 outline 原顺序直接装配”，而是先经过模板重组。
4. 最终落盘是 `templateRenderer.render(...)` 之后一次性 `createDoc(...)`。

这意味着：

1. 当前问题不在“并行落盘打架”。
2. 当前问题在“章节独立起草 + 模板二次分槽重排 + 编号重写逻辑错误”。

## 4. 当前链路已暴露出的根因

### 4.1 根因一：最终结构真相源不唯一

当前至少存在三套“谁决定最终结构”的来源：

1. `DocumentOutline.sections`
2. `List<DocumentSectionDraft>`
3. `TemplateSectionResolver + report-template`

这三者之间没有单一真相源约束，导致最终顺序不稳定。

### 4.2 根因二：章节正文生成上下文不足

每个 section worker 当前只感知：

1. 当前章节 heading
2. 当前章节 keyPoints
3. task 级上下文

它不感知：

1. 前面章节已经写了什么
2. 后面章节准备怎么写
3. 全文术语、风格、编号、边界的冻结约束

这会导致：

1. 重复表达
2. 术语不一致
3. 承接弱
4. 某些章节像“自己写自己的”

### 4.3 根因三：模板渲染阶段擅自重排章节

当前 `report-template.md` 不是“装配 outline”，而是“把语义相近章节塞进固定槽位，然后把剩余章节统一尾插”。

这会直接导致：

1. `活动时间与地点`
2. `活动预算`
3. `节目安排`

这类没有命中 alias 的章节，被整体甩到文档末尾。

### 4.4 根因四：章节编号重写逻辑与模板顶层顺序不一致

当前 `DocumentTemplateRenderer.render(...)` 中为不同语义槽位写死了错误的 sectionPrefix：

1. `background -> 1`
2. `goal -> 2`
3. `principles -> 3`
4. `plan -> 5`
5. `risks -> 6`
6. `owners -> 7`
7. `timeline -> 8`

但模板实际只有六个顶层章节。

这会直接制造：

1. `四、风险` 下出现 `6.1`
2. `五、分工` 下出现 `7.1`
3. `六、时间计划` 下出现 `8.1`

这不是模型偶发，而是确定性的渲染 bug。

### 4.5 根因五：Review 的职责过轻，缺少全文整编阶段

当前 review 更像“内容缺项审查”，而不是“全文一致性编辑”。

它能发现：

1. 缺少必要章节
2. Mermaid 缺失
3. 主题漂移

但它不能可靠解决：

1. 顺序错位
2. 编号错乱
3. 跨章节承接不自然
4. 术语与风格不统一

## 5. 明确不推荐的整改方向

### 5.1 不推荐：回退成单个黑箱节点“一次性写完整篇”

短期看，这似乎可以缓解：

1. 章节乱序
2. 上下文割裂

但这种做法的问题很大：

1. 它会把文档生成重新退化成大模型黑箱。
2. 不利于局部回退。
3. 不利于 Gate 分阶段生效。
4. 不利于任务状态与中间 Artifact 显式沉淀。
5. 不符合 `im_harness架构设计.md` 中“编排对象是步骤图，不是固定黑箱流水线”的核心思想。

### 5.2 不推荐：继续保留章节独立起草，再靠模板 alias 猜测最终顺序

这条路已经证明会持续制造：

1. 顺序漂移
2. 语义误归类
3. 编号错位
4. 尾插剩余章节

### 5.3 不推荐：把“章节乱序”理解成“串行/并行”的问题

当前代码里章节生成是串行的，乱序依旧存在。

这说明真正问题是：

1. 数据模型设计错误
2. 最终装配阶段重排错误

而不是线程模型错误。

## 6. 推荐型架构形态

### 6.1 一句话定义

推荐的 docx 生成链路应定义为：

**一个以 `DocumentPlan` 为单一结构真相源、以 section draft 为局部起草单元、以全文整编节点为一致性收口层、以 deterministic render 为唯一落盘入口、并受 Planner/Orchestrator 与分阶段 Gate 约束的文档装配系统。**

### 6.2 核心原则

1. 不回退为单黑箱节点。
2. 也不允许“章节独立生成后靠模板自由重排”。
3. 允许分步骤生成。
4. 但最终顺序必须只认一份冻结后的 `DocumentPlan`。
5. 最终装配必须是单点、确定性、可验证的。

### 6.3 推荐链路总览

推荐链路：

1. `DocumentPlan` 冻结
2. `SectionDraft` 受约束生成
3. `Consistency / Composition Pass`
4. `Deterministic Render`
5. `Publish Gate`
6. `Doc Tool Write`

其中：

1. 前三步可以生成中间 Artifact。
2. 真正写飞书只能发生在最后一步。
3. 任何中间步骤都不能私自决定最终章节顺序。

## 7. 新的数据模型设计

### 7.1 新增 `DocumentPlan`

`DocumentPlan` 必须成为文档生成链路中的单一结构真相源。

建议字段：

1. `planId`
2. `taskId`
3. `title`
4. `templateType`
5. `orderedSections`
6. `globalStyleGuide`
7. `terminologyRules`
8. `requiredArtifacts`
9. `diagramPlan`
10. `approvalRequirements`
11. `version`

### 7.2 `DocumentPlan.orderedSections`

每一节建议包含：

1. `sectionId`
2. `index`
3. `heading`
4. `purpose`
5. `mustCover`
6. `allowedSubheadings`
7. `dependsOn`
8. `semanticType`
9. `riskLevel`
10. `acceptanceCriteria`

说明：

1. `semanticType` 只作为语义标签，不得再用来决定最终顺序。
2. 顺序只认 `index`。

### 7.3 `SectionDraft`

每一节正文不再只是松散的 `heading + body`，而要带上 plan 锚点：

1. `sectionId`
2. `heading`
3. `body`
4. `inputSummary`
5. `upstreamDependencies`
6. `status`
7. `qualityFlags`
8. `version`

### 7.4 新增 `ComposedDocumentDraft`

这是全文整编后的统一草稿。

建议字段：

1. `taskId`
2. `planId`
3. `orderedSections`
4. `composedMarkdown`
5. `consistencyReport`
6. `version`

## 8. 新的执行链路设计

### 8.1 阶段一：Plan Freeze

职责：

1. 基于用户需求生成 `DocumentPlan`
2. 冻结章节顺序、顶层结构、每节职责、统一术语、风格约束

要求：

1. 这一步通过 `Plan Gate` 后，结构顺序不允许在后续阶段被随意改写。
2. 后续所有 worker 都必须引用同一份 `DocumentPlan`。

### 8.2 阶段二：Section Drafting

职责：

1. 对每个 `sectionId` 生成正文草稿

输入必须包含：

1. 当前 section spec
2. 全量 `DocumentPlan`
3. 已完成章节摘要
4. 全文术语表
5. 全文风格规则

说明：

1. 可以串行。
2. 也可以有限并行。
3. 关键不是并行与否，而是输入必须来自统一 contract。

### 8.3 阶段三：Consistency / Composition Pass

职责：

1. 基于全量 `SectionDraft` 做全文一致性整编

必须覆盖：

1. 去重
2. 术语统一
3. 承接修复
4. 编号合法化
5. Mermaid 与正文对齐
6. 章节边界清理

注意：

1. 这一步是 editor / composer，不是 publish。
2. 它不直接写飞书。

### 8.4 阶段四：Deterministic Render

职责：

1. 按 `DocumentPlan.orderedSections` 顺序，确定性拼装最终 Markdown / XML

红线：

1. renderer 不能再通过 alias 猜顺序。
2. renderer 不能再决定“哪些章节放前面、哪些尾插”。
3. renderer 只能做格式装配，不得做结构重排。

### 8.5 阶段五：Publish Gate

落盘前必须检查：

1. section 顺序是否与 `DocumentPlan` 一致
2. 顶层标题是否完整
3. 编号是否单调合法
4. 是否存在未消费章节
5. 是否存在重复章节
6. 是否存在空章节
7. Mermaid 是否与目标章节一致

未通过则：

1. 不允许写飞书
2. 必须进入 `local replan` 或 `consistency retry`

### 8.6 阶段六：单点写入飞书

真正落盘时：

1. 只允许一个 publish step 执行文档创建 / 覆盖 / 局部写入
2. 不允许多个章节节点直接并发写飞书
3. 所有写入都必须带明确前置 Gate 结果

## 9. 与 `im_harness` 核心思想的关系

### 9.1 为什么这套方案是符合 `im_harness` 的

因为它保留了：

1. `Planner / Orchestrator` 唯一大脑层
2. 步骤图 / 状态机编排
3. 分阶段 Gate
4. 显式 Artifact 沉淀
5. 局部回退优先
6. 能力节点显式输入输出

### 9.2 为什么“全退回单黑箱节点”不符合 `im_harness`

因为它会损失：

1. 中间 Artifact 可见性
2. 分阶段质量门禁
3. 局部回退能力
4. 状态投影能力
5. 可重规划能力

### 9.3 正确理解“单点装配”

本次整改建议的不是：

1. 单节点黑箱生成全文

而是：

1. 分步骤生成
2. 单点整编
3. 单点渲染
4. 单点落盘

这四者要区分清楚。

## 10. 对当前代码的明确改造要求

### 10.1 `DocumentWorkflowNodes`

必须重构为：

1. `generateOutline` 不再只产出松散 outline，要升级为 `DocumentPlan` 生成
2. `generateSections` 必须按 `sectionId` 生成 draft，并带全局 contract
3. `reviewDoc` 不能只做缺项 review，要拆成：
   - content review
   - consistency compose
4. `writeDocAndSync` 只能消费 deterministic render 的结果，不得再混合做结构决策

### 10.2 `TemplateSectionResolver`

必须降级其职责：

1. 可以保留语义标签辅助
2. 但不得再作为顺序决定器
3. 不得再把未命中章节统一尾插到文档末尾

### 10.3 `DocumentTemplateRenderer`

必须改成：

1. 按 `DocumentPlan.orderedSections` 顺序直接渲染
2. sectionPrefix 必须由 plan index 推导，不允许写死错位前缀
3. 模板只能决定壳子，不得重排业务章节

### 10.4 `DocumentBodyNormalizer`

职责保留，但边界收紧：

1. 只做结构正规化
2. 不做语义重排
3. 编号改写必须严格依赖传入的 section index

### 10.5 新增 Gate

至少新增：

1. `Structure Gate`
2. `Ordering Gate`
3. `Heading Consistency Gate`
4. `Draft Completeness Gate`

## 11. 推荐的迁移阶段

### Phase 1：止血

目标：

1. 停止继续产生乱序文档

动作：

1. 移除 alias 驱动的顺序重排
2. 修正 renderer 的错误编号前缀
3. 最终顺序改为直接尊重 outline 原顺序
4. 加入落盘前结构合法性 Gate

### Phase 2：结构真相源升级

目标：

1. 从 `DocumentOutline` 升级到 `DocumentPlan`

动作：

1. 增加 `sectionId`
2. 增加 orderedSections
3. 增加全局 style / terminology contract

### Phase 3：引入 Composer

目标：

1. 解决“章节各写各的”

动作：

1. 新增 `Consistency / Composition Pass`
2. 统一做全文整编
3. review 与 compose 解耦

### Phase 4：有限并行优化

目标：

1. 在保证 contract 不变的前提下，提升章节生成吞吐

动作：

1. 只允许对无依赖章节做有限并行 draft
2. 并行生成不等于并行落盘
3. 仍然必须单点 compose + single publish

## 12. Codex 实施红线

Codex 在实现时必须遵守以下红线：

1. 不允许把整个文档生成退化成单个黑箱大模型调用。
2. 不允许继续保留“模板 alias 猜顺序 + 剩余章节尾插”。
3. 不允许多个章节节点直接写飞书。
4. 不允许 renderer 再决定结构顺序。
5. 不允许 section worker 在没有全局 contract 的情况下独立起草正文。
6. 不允许把 review 继续当作唯一的全文兜底步骤。
7. 不允许引入第二个 supervisor / harness 主脑。
8. 不允许违反 `im_harness架构设计.md` 中“Planner / Orchestrator 唯一大脑层”的原则。

## 13. 交付验收标准

整改完成后，至少应满足：

1. 同一份 outline 在最终文档中的顶层顺序与 `DocumentPlan.orderedSections` 完全一致。
2. 不再出现“模板槽位命中章节在前、剩余章节统一尾插”的现象。
3. 不再出现 `四、风险` 下挂 `6.1`、`五、分工` 下挂 `7.1` 这类错位编号。
4. 每个 section draft 都能追溯到唯一 `sectionId`。
5. 章节生成输入包含全局 contract，而非只包含当前 heading。
6. 飞书写入前有明确的 `Structure Gate` 和 `Publish Gate`。
7. 文档生成链路中的每个主要阶段都能沉淀 Step / Artifact / Event，符合 `im_harness` 的状态投影要求。

---

## 14. 给 Codex 的最终执行口径

如果你是 Codex，请按以下口径执行：

1. 这不是一次局部补丁任务，而是一次文档生成链路的结构性整改任务。
2. 你必须先把 `DocumentPlan` 设计成新的单一结构真相源。
3. 你必须把当前“章节独立起草 + 模板二次重排”的链路重构为“计划冻结 -> 局部起草 -> 全文整编 -> 确定性渲染 -> 单点写入”。
4. 你不得简单回退成单节点整文黑箱生成。
5. 你必须保证整改后的链路仍然符合 `im_harness架构设计.md` 的核心思想：状态机、事件流、唯一大脑层、局部回退、分阶段 Gate、Artifact 沉淀。
6. 你在实现时，优先修复结构真相源、渲染顺序、编号逻辑，再处理并行优化问题。顺序问题未解决前，不要讨论并行性能。

一句话总结：

**正确方向不是“更多章节节点”也不是“退回单个黑箱节点”，而是“用全局 contract 约束局部生成，并用单点装配节点收口最终结构”。**
